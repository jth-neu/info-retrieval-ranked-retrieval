<h1 id="firstHeading" class="firstHeading" lang="en">Link rot</h1>			<div id="bodyContent" class="mw-body-content">
				<div id="siteSub" class="noprint">From Wikipedia, the free encyclopedia</div>				<div id="contentSub"><span class="mw-redirectedfrom">  (Redirected from <a href="/w/index.php?title=Linkrot&amp;redirect=no" class="mw-redirect" title="Linkrot">Linkrot</a>)</span></div>
				<div id="jump-to-nav"></div>				<a class="mw-jump-link" href="#mw-head">Jump to navigation</a>
				<a class="mw-jump-link" href="#p-search">Jump to search</a>
				<div id="mw-content-text" lang="en" dir="ltr" class="mw-content-ltr"><div class="mw-parser-output"><div role="note" class="hatnote navigation-not-searchable"><span class="plainlinks selfreference noprint"> For link rot in Wikipedia, see <a href="/wiki/Wikipedia:Link_rot" title="Wikipedia:Link rot">Wikipedia:Link rot</a>.</span></div>
<p><b>Link rot</b> (or <b>linkrot</b>) is the process by which <a href="/wiki/Hyperlink" title="Hyperlink">hyperlinks</a> on individual <a href="/wiki/Website" title="Website">websites</a> or the <a href="/wiki/Internet" title="Internet">Internet</a> in general point to <a href="/wiki/Web_page" title="Web page">web pages</a>, <a href="/wiki/Server_(computing)" title="Server (computing)">servers</a> or other resources that have become permanently unavailable. The phrase also describes the effects of failing to update out-of-date <a href="/wiki/Web_page" title="Web page">web pages</a> that clutter <a href="/wiki/Search_engine" class="mw-redirect" title="Search engine">search engine</a> results. Research<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">&#91;1&#93;</a></sup><sup id="cite_ref-2" class="reference"><a href="#cite_note-2">&#91;2&#93;</a></sup> shows that the <a href="/wiki/Half-life" title="Half-life">half-life</a> of a random webpage is two years.
</p>

<h2><span class="mw-headline" id="Terminology">Terminology</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Link_rot&amp;action=edit&amp;section=1" title="Edit section: Terminology">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Link rot is also called "link death", "link breaking" or "reference rot". A link that does not work any more is called a "broken link", "dead link", or "dangling link". Formally, this is a form of <a href="/wiki/Dangling_reference" class="mw-redirect" title="Dangling reference">dangling reference</a>: The target of the reference no longer exists.
</p>
<h2><span class="mw-headline" id="Causes">Causes</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Link_rot&amp;action=edit&amp;section=2" title="Edit section: Causes">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>One of the most common reasons for a broken link is that the web page to which it points no longer exists. This frequently results in a <a href="/wiki/HTTP_404" title="HTTP 404">404 error</a>, which indicates that the web server responded but the specific page could not be found. Another type of dead link occurs when the server that hosts the target page stops working or relocates to a new <a href="/wiki/Domain_name" title="Domain name">domain name</a>. The browser may return a <a href="/wiki/Domain_Name_System" title="Domain Name System">DNS</a> error or display a site unrelated to the content originally sought. The latter can occur when a domain name lapses and is reregistered by another party. Other reasons for broken links include:
</p>
<ul><li>Websites can be restructured or redesigned, or the underlying technology can be changed, altering or invalidating large numbers of inbound or internal links.</li>
<li>Many news sites keep articles freely accessible for only a short time period, and then move them behind a <a href="/wiki/Paywall" title="Paywall">paywall</a>. This causes a significant loss of supporting links in sites discussing news events and using media sites as references.</li>
<li>Links may expire<sup class="noprint Inline-Template" style="margin-left:0.1em; white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:Please_clarify" title="Wikipedia:Please clarify"><span title="domain registration? (August 2018)">clarification needed</span></a></i>&#93;</sup>.</li>
<li>Content may be intentionally removed by the owner.</li>
<li>A server may be upgraded and code (e.g. PHP) may no longer function properly.</li>
<li>Links may be removed as a result of legal action or court order.</li>
<li>Search results from social media such as <a href="/wiki/Facebook" title="Facebook">Facebook</a> and <a href="/wiki/Tumblr" title="Tumblr">Tumblr</a> are prone to link rot because of frequent changes in user privacy, the deletion of accounts, search result pointing to a dynamic page that has new results that differ from the <a href="/wiki/Cache_(computing)" title="Cache (computing)">cached result</a>, or the deletion of links or photos.</li>
<li>Links can contain ephemeral, user-specific information such as session or login data. Because these are not universally valid, the result can be a broken link.</li>
<li>A link might be broken because of some form of blocking such as <a href="/wiki/Content_filter" class="mw-redirect" title="Content filter">content filters</a> or <a href="/wiki/Firewall_(networking)" class="mw-redirect" title="Firewall (networking)">firewalls</a>.</li>
<li>A website may be closed or taken down, invalidating the links which are pointing to it.</li>
<li>A website might change its domain name. Links pointing to the old name might then become invalid.</li>
<li>Dead links can occur on the authoring side, when website content is assembled from Internet sources and deployed without properly verifying the link targets.</li>
<li>As new private <a href="/wiki/Generic_top-level_domain" title="Generic top-level domain">gTLDs</a> became popular, top-level-domains like <code>.mcdonalds</code> or <code>.xperia</code> were revoked.<sup id="cite_ref-3" class="reference"><a href="#cite_note-3">&#91;3&#93;</a></sup></li></ul>
<h2><span class="mw-headline" id="Prevalence">Prevalence</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Link_rot&amp;action=edit&amp;section=3" title="Edit section: Prevalence">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The <a href="/wiki/HTTP_404" title="HTTP 404">404 "Not Found"</a> response is familiar to even the occasional web user.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (May 2018)">citation needed</span></a></i>&#93;</sup> A number of studies have examined the prevalence of link rot on the web, in academic literature, and in <a href="/wiki/Digital_library" title="Digital library">digital libraries</a>.<sup id="cite_ref-Habibzadeh_4-0" class="reference"><a href="#cite_note-Habibzadeh-4">&#91;4&#93;</a></sup> In a 2003 experiment, Fetterly et al. discovered that about one link out of every 200 disappeared each week from the Internet. McCown et al. (2005) discovered that half of the <a href="/wiki/Uniform_Resource_Locator" class="mw-redirect" title="Uniform Resource Locator">URLs</a> cited in <a href="/wiki/D-Lib_Magazine" title="D-Lib Magazine">D-Lib Magazine</a> articles were no longer accessible 10 years after publication, and other studies have shown link rot in academic literature to be even worse (Spinellis, 2003, Lawrence et al., 2001). Nelson and Allen (2002) examined link rot in digital libraries and found that about 3% of the objects were no longer accessible after one year. In 2014, bookmarking site <a href="/wiki/Pinboard_(website)" title="Pinboard (website)">Pinboard</a>'s owner <a href="/wiki/Maciej_Ceg%C5%82owski" title="Maciej Cegłowski">Maciej Cegłowski</a> reported a “pretty steady rate” of 5% link rot per year.<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">&#91;5&#93;</a></sup>
</p><p>A 2014 <a href="/wiki/Harvard_Law_School" title="Harvard Law School">Harvard Law School</a> study by <a href="/wiki/Jonathan_Zittrain" title="Jonathan Zittrain">Jonathan Zittrain</a>, Kendra Albert and <a href="/wiki/Lawrence_Lessig" title="Lawrence Lessig">Lawrence Lessig</a>, determined that approximately 50% of the URLs in U.S. Supreme Court opinions no longer link to the original information.<sup id="cite_ref-permacc_6-0" class="reference"><a href="#cite_note-permacc-6">&#91;6&#93;</a></sup> They also found that in a selection of legal journals published between 1999 and 2011, more than 70% of the links no longer functioned as intended. A 2013 study in <i><a href="/wiki/BMC_Bioinformatics" title="BMC Bioinformatics">BMC Bioinformatics</a></i> analyzed nearly 15,000 links in abstracts from <a href="/wiki/Thomson_Reuters" title="Thomson Reuters">Thomson Reuters</a>’ Web of Science citation index and found that the median lifespan of web pages was 9.3 years, and just 62% were archived.<sup id="cite_ref-7" class="reference"><a href="#cite_note-7">&#91;7&#93;</a></sup> In August 2015 Weblock analyzed more than 180,000 links from references in the full-text corpora of three major open access publishers and found that overall 24.5% of links cited were no longer available.<sup id="cite_ref-8" class="reference"><a href="#cite_note-8">&#91;8&#93;</a></sup>
</p>
<h2><span class="mw-headline" id="Discovering">Discovering</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Link_rot&amp;action=edit&amp;section=4" title="Edit section: Discovering">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Discovering broken links might be done manually or automatically. Automated methods, including <a href="/wiki/Plug-in_(computing)" title="Plug-in (computing)">plug-ins</a> for <a href="/wiki/WordPress" title="WordPress">WordPress</a>, <a href="/wiki/Drupal" title="Drupal">Drupal</a> and other <a href="/wiki/Content_management_systems" class="mw-redirect" title="Content management systems">content management system</a> can be used to detect the presence of broken <a href="/wiki/Uniform_Resource_Locator" class="mw-redirect" title="Uniform Resource Locator">URLs</a>. An alternative is using a specific broken link checker like <a href="/wiki/Xenu%27s_Link_Sleuth" title="Xenu&#39;s Link Sleuth">Xenu's Link Sleuth</a>. However, if a URL returns an <a href="/wiki/HTTP" class="mw-redirect" title="HTTP">HTTP</a> <a href="/wiki/200_OK" class="mw-redirect" title="200 OK">200 (OK)</a> response, it may be accessible, but the contents of the page could have changed and may no longer be relevant. So manual checking links seems to be a must. Some web servers also return a <a href="/wiki/Soft_404" class="mw-redirect" title="Soft 404">soft 404</a>, reporting to computers that the link works even though it doesn't. Bar-Yossef et al. (2004) <sup id="cite_ref-9" class="reference"><a href="#cite_note-9">&#91;9&#93;</a></sup> developed a heuristic for automatically discovering soft 404s.
</p>
<h2><span class="mw-headline" id="Combating">Combating</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Link_rot&amp;action=edit&amp;section=5" title="Edit section: Combating">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>There are numerous solutions for tackling broken links: Some work to prevent them in the first place, while others try to resolve them when they have occurred. There are also numerous tools that have been developed to help combat link rot.
</p>
<h3><span class="mw-headline" id="Authoring">Authoring</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Link_rot&amp;action=edit&amp;section=6" title="Edit section: Authoring">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<ul><li>Carefully select and implement hyperlinks, and verify them regularly after publication. Best practices include linking to primary rather than secondary sources and prioritizing stable sites. McCown et al., 2005, suggests avoiding URL citations that point to resources on researchers' personal pages.</li>
<li>Always look for the most compact and direct URL available, and ensure that it’s <a href="/wiki/Clean_URL" title="Clean URL">clean</a>, with no unnecessary information after the core of the URL.<sup id="cite_ref-jrlinkrot_10-0" class="reference"><a href="#cite_note-jrlinkrot-10">&#91;10&#93;</a></sup> This process is often referred to as <a href="/wiki/URL_normalization" title="URL normalization">URL normalization</a> or <a href="/wiki/URL_canonicalization" class="mw-redirect" title="URL canonicalization">URL canonicalization</a>.</li>
<li>Whenever possible, use <a href="/wiki/Persistent_identifier" title="Persistent identifier">persistent identifiers</a> (URLs designed for durability) such as ARKs, <a href="/wiki/Digital_object_identifier" title="Digital object identifier">DOIs</a>, Handle System references, and PURLs.</li>
<li>Avoid linking to <a href="/wiki/PDF" title="PDF">PDF</a> documents if possible. Because PDFs are documents rather than web pages, their content can change without notice, and their names are more likely to contain characters such as spaces that must be translated into safe codes for URLs. Large PDFs may also download slowly and cause a timeout error.<sup id="cite_ref-jrlinkrot_10-1" class="reference"><a href="#cite_note-jrlinkrot-10">&#91;10&#93;</a></sup></li>
<li>Avoid linking to pages deep in a website, a practice known as <a href="/wiki/Deep_linking" title="Deep linking">deep linking</a>.</li>
<li>Use web archiving services (for example, <a href="/wiki/WebCite" title="WebCite">WebCite</a>) to permanently archive and retrieve cited Internet references (Eysenbach and Trudel, 2005).</li></ul>
<h3><span class="mw-headline" id="Server_side">Server side</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Link_rot&amp;action=edit&amp;section=7" title="Edit section: Server side">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<ul><li>Never change URLs and never remove pages. If there is a reason to no longer have a page, such as a news site redacting a story, replace it with a message explaining its removal.</li>
<li>When URLs change, use <a href="/wiki/URL_redirection" title="URL redirection">redirection</a> mechanisms such as "<a href="/wiki/HTTP_301" title="HTTP 301">301: Moved Permanently</a>" to automatically refer browsers and crawlers to the new location.</li>
<li><a href="/wiki/Web_content_management_system" title="Web content management system">Content management systems</a> may offer built-in solutions to the management of links, such as updating them when content is changed or moved on a site.</li>
<li><a href="/wiki/WordPress" title="WordPress">WordPress</a> guards against link rot by replacing non-canonical URLs with their <a href="/wiki/URL_canonicalization" class="mw-redirect" title="URL canonicalization">canonical</a> versions.<sup id="cite_ref-Justaddwater_2007_11-0" class="reference"><a href="#cite_note-Justaddwater_2007-11">&#91;11&#93;</a></sup></li>
<li><a href="/wiki/IBM" title="IBM">IBM</a>'s <a href="/wiki/IBM_Peridot_(software)" title="IBM Peridot (software)">Peridot</a> attempts to automatically fix broken links.</li>
<li><a href="/wiki/Permalink" title="Permalink">Permalinking</a> stops broken links by guaranteeing that the content will not move for the foreseeable future. Another form of permalinking is linking to a permalink that then redirects to the actual content, ensuring that even though the real content may be moved etc., links pointing to the resources stay intact.</li>
<li>Design URLs—for example, <a href="/wiki/Semantic_URL" class="mw-redirect" title="Semantic URL">semantic URLs</a>—such that they won't need to change when a different person takes over maintenance of a document or when different software is used on the server.<sup id="cite_ref-12" class="reference"><a href="#cite_note-12">&#91;12&#93;</a></sup></li></ul>
<h3><span class="mw-headline" id="User_side">User side</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Link_rot&amp;action=edit&amp;section=8" title="Edit section: User side">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<ul><li>The Linkgraph widget gets the URL of the correct page based upon the old broken URL by using historical location information.</li>
<li>The Google 404 Widget attempts to "guess" the correct URL, and also provides the user with a search box to find the correct page.</li>
<li>When a user receives a 404 response, the <a href="/wiki/Google_Toolbar" title="Google Toolbar">Google Toolbar</a> attempts to assist the user in finding the missing page.<sup id="cite_ref-GoogleToolbar_13-0" class="reference"><a href="#cite_note-GoogleToolbar-13">&#91;13&#93;</a></sup></li></ul>
<h3><span class="mw-headline" id="Web_archiving">Web archiving</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Link_rot&amp;action=edit&amp;section=9" title="Edit section: Web archiving">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Web_archiving" title="Web archiving">Web archiving</a></div>
<p>To combat link rot, <a href="/wiki/Web_archiving" title="Web archiving">web archivists</a> are actively engaged in collecting the <a href="/wiki/World_Wide_Web" title="World Wide Web">Web</a> or particular portions of the Web and ensuring the collection is <a href="/wiki/Digital_preservation" title="Digital preservation">preserved</a> in an <a href="/wiki/Archive" title="Archive">archive</a>, such as an <a href="/wiki/Archive_site" title="Archive site">archive site</a>, for future researchers, historians, and the public. The goal of the <a href="/wiki/Internet_Archive" title="Internet Archive">Internet Archive</a> is to maintain an archive of the entire Web, taking periodic snapshots of pages that can then be accessed for free via the <a href="/wiki/Wayback_Machine" title="Wayback Machine">Wayback Machine</a>. In January 2013 the company announced that it had reached the milestone of 240 billion archived URLs.<sup id="cite_ref-14" class="reference"><a href="#cite_note-14">&#91;14&#93;</a></sup> <a href="/wiki/National_library" title="National library">National libraries</a>, <a href="/wiki/National_archive" class="mw-redirect" title="National archive">national archives</a> and other organizations are also involved in archiving culturally important Web content.
</p><p>Individuals may use a number of tools that allow them to archive web resources that may go missing in the future:
</p>
<ul><li>The "WayBack Machine", at the <a href="/wiki/Internet_Archive" title="Internet Archive">Internet Archive</a>,<sup id="cite_ref-15" class="reference"><a href="#cite_note-15">&#91;15&#93;</a></sup> is a free website that archives old web pages. It does not archive websites whose owners have stated they do not want their website archived.</li>
<li><a href="/wiki/WebCite" title="WebCite">WebCite</a>, a tool specifically for scholarly authors, journal editors and publishers to permanently archive "on-demand" and retrieve cited Internet references (Eysenbach and Trudel, 2005).</li>
<li><a href="/wiki/Archive.is" title="Archive.is">Archive.is</a>, an archive site which stores snapshots of web pages. It retrieves one page at a time, but unlike WebCite, it includes <a href="/wiki/Web_2.0" title="Web 2.0">Web 2.0</a> sites such as Google Maps and Twitter.</li>
<li><a href="/wiki/Perma.cc" title="Perma.cc">Perma.cc</a>, which is supported by the Harvard Law School together with a broad coalition of university libraries, takes a snapshot of a URL's content and returns a permanent link.<sup id="cite_ref-permacc_6-1" class="reference"><a href="#cite_note-permacc-6">&#91;6&#93;</a></sup></li>
<li>The Hiberlink project, a collaboration between the University of Edinburgh, the Los Alamos National Laboratory and others, is working to measure “reference rot” in online academic articles, and also to what extent Web content has been archived.<sup id="cite_ref-16" class="reference"><a href="#cite_note-16">&#91;16&#93;</a></sup> A related project, <a href="/wiki/Memento_Project" title="Memento Project">Memento</a>, has established a technical standard for accessing online content as it existed in the past.<sup id="cite_ref-17" class="reference"><a href="#cite_note-17">&#91;17&#93;</a></sup></li>
<li>Some <a href="/wiki/Social_bookmarking" title="Social bookmarking">social bookmarking</a> websites allow users to make online clones of any web page on the internet, creating a copy at an independent url which remains online even if the original page goes down.</li>
<li>Amber, created by the <a href="/wiki/Harvard_University" title="Harvard University">Harvard</a> <a href="/wiki/Berkman_Center_for_Internet_%26_Society" class="mw-redirect" title="Berkman Center for Internet &amp; Society">Berkman Center</a>, is a tool built to fight link rot through archiving links on <a href="/wiki/WordPress" title="WordPress">Wordpress</a> and <a href="/wiki/Drupal" title="Drupal">Drupal</a> sites to prevent web censorship and bolster content preservation.<sup id="cite_ref-18" class="reference"><a href="#cite_note-18">&#91;18&#93;</a></sup></li></ul>
<p>However, such preserving systems may encounter on and off service interruption so that the preserved URLs are intermittently unavailable.<sup id="cite_ref-19" class="reference"><a href="#cite_note-19">&#91;19&#93;</a></sup>
</p>
