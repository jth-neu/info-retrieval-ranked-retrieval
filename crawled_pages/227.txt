<h1 id="firstHeading" class="firstHeading" lang="en">Form (HTML)</h1>			<div id="bodyContent" class="mw-body-content">
				<div id="siteSub" class="noprint">From Wikipedia, the free encyclopedia</div>				<div id="contentSub"><span class="mw-redirectedfrom">  (Redirected from <a href="/w/index.php?title=Web_form&amp;redirect=no" class="mw-redirect" title="Web form">Web form</a>)</span></div>
				<div id="jump-to-nav"></div>				<a class="mw-jump-link" href="#mw-head">Jump to navigation</a>
				<a class="mw-jump-link" href="#p-search">Jump to search</a>
				<div id="mw-content-text" lang="en" dir="ltr" class="mw-content-ltr"><div class="mw-parser-output"><p>A <b>webform</b>, <b>web form</b> or <b>HTML form</b> on a <a href="/wiki/Web_page" title="Web page">web page</a> allows a user to enter data that is sent to a <a href="/wiki/Server_(computing)" title="Server (computing)">server</a> for processing. Forms can resemble <a href="/wiki/Form_(document)" title="Form (document)">paper</a> or <a href="/wiki/Database" title="Database">database</a> forms because web users fill out the forms using checkboxes, <a href="/wiki/Radio_buttons" class="mw-redirect" title="Radio buttons">radio buttons</a>, or <a href="/wiki/Text_fields" class="mw-redirect" title="Text fields">text fields</a>.  For example, forms can be used to enter <a href="/wiki/Shipping" class="mw-redirect" title="Shipping">shipping</a> or <a href="/wiki/Credit_card" title="Credit card">credit card</a> data to order a product, or can be used to retrieve search results from a <a href="/wiki/Search_engine" class="mw-redirect" title="Search engine">search engine</a>.
</p>

<h2><span class="mw-headline" id="Description">Description</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Form_(HTML)&amp;action=edit&amp;section=1" title="Edit section: Description">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="thumb tright"><div class="thumbinner" style="width:395px;"><a href="/wiki/File:Sample_web_form.png" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/3/34/Sample_web_form.png" width="393" height="320" class="thumbimage" data-file-width="393" data-file-height="320" /></a>  <div class="thumbcaption"><div class="magnify"><a href="/wiki/File:Sample_web_form.png" class="internal" title="Enlarge"></a></div>Sample form. The form is enclosed in an <a href="/wiki/HTML_table" class="mw-redirect" title="HTML table">HTML table</a> for visual layout.</div></div></div>
<p>Forms are enclosed in the HTML <code>form</code> tag. This tag specifies the <a href="/wiki/Communication_endpoint" title="Communication endpoint">communication endpoint</a> the data entered into the form should be submitted to, and the method of submitting the data, GET or POST.
</p>
<h3><span class="mw-headline" id="Elements">Elements</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Form_(HTML)&amp;action=edit&amp;section=2" title="Edit section: Elements">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Forms can be made up of standard <a href="/wiki/Graphical_user_interface" title="Graphical user interface">graphical user interface</a> elements:
</p>
<ul><li>text — a simple <a href="/wiki/Text_box" title="Text box">text box</a> that allows input of a single line of text.</li>
<li>email - a type of text that requires a partially validated email address</li>
<li>number - a type of text that requires a number</li>
<li>password — similar to text, it is used for security purposes, in which the characters typed in are invisible or replaced by symbols such as *)</li>
<li>radio — a <a href="/wiki/Radio_button" title="Radio button">radio button</a></li>
<li>file — a <a href="/wiki/File_select" title="File select">file select</a> control for uploading a file</li>
<li>reset — a <a href="/wiki/Reset_button" title="Reset button">reset button</a> that, when activated, tells the browser to restore the values to their initial values.</li>
<li>submit — a <a href="/wiki/Button_(computing)" title="Button (computing)">button</a> that tells the browser to take action on the form (typically to send it to a server)</li>
<li>textarea — much like the text input field except a textarea allows for multiple rows of data to be shown and entered</li>
<li>select — a <a href="/wiki/Drop-down_list" title="Drop-down list">drop-down list</a> that displays a list of items a user can select from</li></ul>
<p>The sample image on the right shows most of these elements:
</p>
<ul><li>a text box asking for your name</li>
<li>a pair of radio buttons asking you to pick your sex</li>
<li>a <a href="/wiki/Drop-down_list" title="Drop-down list">select box</a> giving you a list of eye colors to choose from</li>
<li>a pair of check boxes to click on if they apply to you</li>
<li>a text area to describe your athletic ability</li>
<li>a submit button to send it to the server</li></ul>
<p>These basic elements provide most common <a href="/wiki/Graphical_user_interface" title="Graphical user interface">graphical user interface</a> (GUI) elements, but not all. For example, there are no equivalents to a <a href="/wiki/Tree_view" title="Tree view">tree view</a> or <a href="/wiki/Grid_view" title="Grid view">grid view</a>.
</p><p>A grid view, however, can be mimicked by using a standard HTML <a href="/wiki/Table_(HTML)" class="mw-redirect" title="Table (HTML)">table</a> with each cell containing a text input element. A tree view could also be mimicked through nested tables or, more <a href="/wiki/HTML#Semantic_HTML" title="HTML">semantically</a> appropriately, nested <a href="/wiki/HTML_element#Lists" title="HTML element">lists</a>. In both cases, a server-side process is responsible for processing the information, while JavaScript handles the user-interaction. Implementations of these interface elements are available through <a href="/wiki/JavaScript_library" title="JavaScript library">JavaScript libraries</a> such as <a href="/wiki/JQuery" title="JQuery">jQuery</a>.
</p><p>HTML 4 introduced the <code>label</code> tag, which is intended to represent a caption in a user interface, and can be associated with a specific form control by specifying the <code>id</code> attribute of the control in the label tag's <code>for</code> attribute.<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">&#91;1&#93;</a></sup> This allows labels to stay with their elements when a window is resized and to allow more desktop-like functionality e.g. clicking a radio button or checkbox's label will activate the associated input element.
</p><p>HTML 5 introduces a number of input tags that can be represented by other interface elements. Some are based upon text input fields and are intended to input and validate specific common data. These include <code>email</code> to enter email addresses, <code>tel</code> for telephone numbers, <code>number</code> for numeric values. There are additional attributes to specify required fields, fields that should have keyboard focus when the web page containing the form is loaded, and placeholder text that is displayed within the field but is not user input (such as the 'Search' text displayed in many search input fields before a search term is entered.) These tasks used to be handled with <a href="/wiki/JavaScript" title="JavaScript">JavaScript</a>, but had become so common that support for them was added to the standard. The <code>date</code> input type displays a calendar from which the user can select a date or date range.<sup id="cite_ref-2" class="reference"><a href="#cite_note-2">&#91;2&#93;</a></sup><sup id="cite_ref-3" class="reference"><a href="#cite_note-3">&#91;3&#93;</a></sup> And the <code>color</code> input type can be represented as an input text simply checking the value entered is a correct hexadecimal representation of a color, according to the specification,<sup id="cite_ref-4" class="reference"><a href="#cite_note-4">&#91;4&#93;</a></sup> or a color picker widget (the latter being the solution used in most browsers which support this attribute).
</p>
<h2><span class="mw-headline" id="Submission">Submission</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Form_(HTML)&amp;action=edit&amp;section=3" title="Edit section: Submission">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>When data that has been entered into HTML forms is submitted, the names and values in the form elements are encoded and sent to the server in an <a href="/wiki/HTTP" class="mw-redirect" title="HTTP">HTTP</a> request message using <a href="/wiki/Hypertext_Transfer_Protocol#Request_methods" title="Hypertext Transfer Protocol">GET</a> or <a href="/wiki/POST_(HTTP)" title="POST (HTTP)">POST</a>. Historically, an <a href="/wiki/Email" title="Email">email</a> transport was also used.<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">&#91;5&#93;</a></sup> The default <a href="/wiki/Mime_type" class="mw-redirect" title="Mime type">mime type</a>, <a href="/wiki/Internet_media_type" class="mw-redirect" title="Internet media type">Internet media type</a> <a href="/wiki/Application/x-www-form-urlencoded" class="mw-redirect" title="Application/x-www-form-urlencoded">application/x-www-form-urlencoded</a>, is based on a very early version of the general URI <a href="/wiki/Percent-encoding" title="Percent-encoding">percent-encoding</a> rules, with a number of modifications such as <a href="/wiki/Newline" title="Newline">newline</a> normalization and replacing spaces with "<code>+</code>" instead of "<code>%20</code>". Another possible encoding, Internet media type <a href="/wiki/Multipart/form-data" class="mw-redirect" title="Multipart/form-data">multipart/form-data</a>, is also available and is common for POST-based file submissions.
</p>
<h2><span class="mw-headline" id="Use_with_programming_languages">Use with programming languages</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Form_(HTML)&amp;action=edit&amp;section=4" title="Edit section: Use with programming languages">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Forms are usually combined with programs written in various <a href="/wiki/Programming_language" title="Programming language">programming language</a> to allow <a href="/wiki/Software_developer" class="mw-redirect" title="Software developer">developers</a> to create dynamic <a href="/wiki/Web_site" class="mw-redirect" title="Web site">web sites</a>. The most popular languages include both client-side and/or server-side languages.
</p><p>Although any programming language can be used on the server to process a form's data, the most commonly used languages are <a href="/wiki/Scripting_languages" class="mw-redirect" title="Scripting languages">scripting languages</a>, which tend to have stronger <a href="/wiki/String_(programming)" class="mw-redirect" title="String (programming)">string</a> handling functionality than programming languages such as C, and also have automatic memory management which helps to prevent <a href="/wiki/Buffer_overrun" class="mw-redirect" title="Buffer overrun">buffer overrun</a> attacks.
</p>
<h3><span class="mw-headline" id="Client-side">Client-side</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Form_(HTML)&amp;action=edit&amp;section=5" title="Edit section: Client-side">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The <a href="/wiki/De_facto_standard" title="De facto standard"><i>de facto</i></a> <a href="/wiki/Client-side_scripting" class="mw-redirect" title="Client-side scripting">client-side scripting</a> language for web sites is <a href="/wiki/JavaScript" title="JavaScript">JavaScript</a>.
Using JavaScript on the <a href="/wiki/Document_Object_Model" title="Document Object Model">Document Object Model</a> (DOM) leads to the method of <a href="/wiki/Dynamic_HTML" title="Dynamic HTML">Dynamic HTML</a> that allows dynamic creation and modification of a web page within the browser.
</p><p>While client-side languages used in conjunction with forms are limited, they often can serve to do pre-<a href="/wiki/Data_validation" title="Data validation">validation</a> of the form data and/or to prepare the form data to send to a server-side program. This usage is being replaced, however, by <a href="/wiki/HTML5" title="HTML5">HTML5</a>'s new <tt>input</tt> field types and <tt>required</tt> attribute.
</p>
<h3><span class="mw-headline" id="Server-side_execution">Server-side execution</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Form_(HTML)&amp;action=edit&amp;section=6" title="Edit section: Server-side execution">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Server-side code can do a vast assortment of tasks to create dynamic web sites that, for technical or security reasons, client-side code cannot &#8212; from <a href="/wiki/Authentication" title="Authentication">authenticating</a> a <a href="/wiki/Login" title="Login">login</a>, to retrieving and storing data in a <a href="/wiki/Database" title="Database">database</a>, to <a href="/wiki/Spell_checker" title="Spell checker">spell checking</a>, to sending <a href="/wiki/E-mail" class="mw-redirect" title="E-mail">e-mail</a>. A significant advantage to server-side over client-side execution is the concentration of functionality onto the server rather than relying on different <a href="/wiki/Web_browser" title="Web browser">web browsers</a> to implement various functions in consistent, <a href="/wiki/Web_standards" title="Web standards">standardized</a> ways. In addition, processing forms on a server often results in increased security if server-side execution is designed not to trust the data supplied by the client and includes such techniques as <a href="/wiki/HTML_sanitization" title="HTML sanitization">HTML sanitization</a>. One disadvantage to server side code is <a href="/wiki/Scalability" title="Scalability">scalability</a>&#8212;server side processing for all users occurs on the server, while client side processing occurs on individual client computers.
</p>

<h3><span class="mw-headline" id="Interpreted_languages">Interpreted languages</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Form_(HTML)&amp;action=edit&amp;section=7" title="Edit section: Interpreted languages">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Some of the <a href="/wiki/Interpreted_language" title="Interpreted language">interpreted languages</a> commonly used to design interactive forms in web development are <a href="/wiki/PHP" title="PHP">PHP</a>, <a href="/wiki/Python_(programming_language)" title="Python (programming language)">Python</a>, <a href="/wiki/Ruby_(programming_language)" title="Ruby (programming language)">Ruby</a>, <a href="/wiki/Perl" title="Perl">Perl</a>, <a href="/wiki/JavaServer_Pages" title="JavaServer Pages">JSP</a>, <a href="/wiki/Adobe_ColdFusion" title="Adobe ColdFusion">Adobe ColdFusion</a> and some of the compiled languages commonly used are <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a> and <a href="/wiki/C_Sharp_(programming_language)" title="C Sharp (programming language)">C#</a> with <a href="/wiki/ASP.NET" title="ASP.NET">ASP.NET</a>.
</p>
<h4><span class="mw-headline" id="PHP">PHP</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Form_(HTML)&amp;action=edit&amp;section=8" title="Edit section: PHP">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p><a href="/wiki/PHP" title="PHP">PHP</a> is one very common language used for server-side "programming" and is one of the few languages created specifically for <a href="/wiki/Web_programming" class="mw-redirect" title="Web programming">web programming</a>.<sup id="cite_ref-6" class="reference"><a href="#cite_note-6">&#91;6&#93;</a></sup><sup id="cite_ref-7" class="reference"><a href="#cite_note-7">&#91;7&#93;</a></sup>
</p><p>To use PHP with an HTML form, the URL of the PHP script is specified in the <code>action</code> attribute of the form tag. The target PHP file then accesses the data passed by the form through PHP's <code>$_POST</code> or <code>$_GET</code> variables, depending on the value of the <code>method</code> attribute used in the form. Here is a basic form handler PHP script that will display the contents of the "firstname" input field on the page:
</p><p><b>form.html</b>
</p>
<div class="mw-highlight mw-content-ltr" dir="ltr"><pre><span></span><span class="lineno">1 </span><span class="p">&lt;</span><span class="nt">html</span><span class="p">&gt;</span>
<span class="lineno">2 </span><span class="p">&lt;</span><span class="nt">body</span><span class="p">&gt;</span>
<span class="lineno">3 </span> <span class="p">&lt;</span><span class="nt">form</span> <span class="na">action</span><span class="o">=</span><span class="s">&quot;form_handler.php&quot;</span> <span class="na">method</span><span class="o">=</span><span class="s">&quot;GET&quot;</span><span class="p">&gt;</span>
<span class="lineno">4 </span>   User Name: <span class="p">&lt;</span><span class="nt">input</span> <span class="na">name</span><span class="o">=</span><span class="s">&quot;firstname&quot;</span> <span class="na">type</span><span class="o">=</span><span class="s">&quot;text&quot;</span> <span class="p">/&gt;</span>
<span class="lineno">5 </span>   <span class="p">&lt;</span><span class="nt">input</span> <span class="na">type</span><span class="o">=</span><span class="s">&quot;submit&quot;</span> <span class="na">value</span><span class="o">=</span><span class="s">&quot;Submit&quot;</span> <span class="p">/&gt;</span>
<span class="lineno">6 </span> <span class="p">&lt;/</span><span class="nt">form</span><span class="p">&gt;</span>
<span class="lineno">7 </span><span class="p">&lt;/</span><span class="nt">body</span><span class="p">&gt;</span>
<span class="lineno">8 </span><span class="p">&lt;/</span><span class="nt">html</span><span class="p">&gt;</span>
</pre></div>
<p><b>form_handler.php</b>
</p>
<div class="mw-highlight mw-content-ltr" dir="ltr"><pre><span></span><span class="p">&lt;</span><span class="nt">html</span><span class="p">&gt;</span>
<span class="p">&lt;</span><span class="nt">body</span><span class="p">&gt;</span>
<span class="p">&lt;</span><span class="nt">h1</span><span class="p">&gt;</span>User Greeting<span class="p">&lt;/</span><span class="nt">h1</span><span class="p">&gt;</span>
<span class="cp">&lt;?php</span>
 <span class="c1">// This will print whatever the user entered into the form.html page.</span>
 <span class="nv">$name</span> <span class="o">=</span> <span class="nb">filter_input</span><span class="p">(</span><span class="nx">INPUT_GET</span><span class="p">,</span> <span class="s1">&#39;firstname&#39;</span><span class="p">,</span> <span class="nx">FILTER_SANITIZE_STRING</span><span class="p">);</span>
 <span class="k">echo</span> <span class="s2">&quot;Hello, &quot;</span><span class="o">.</span> <span class="nv">$name</span> <span class="o">.</span><span class="s2">&quot;!&quot;</span><span class="p">;</span>
<span class="cp">?&gt;</span>
<span class="p">&lt;/</span><span class="nt">body</span><span class="p">&gt;</span>
<span class="p">&lt;/</span><span class="nt">html</span><span class="p">&gt;</span>
</pre></div>
<p>The sample code above  uses PHP's <code>filter_input()</code> function to sanitize the user's input before inserting it onto the page. Simply printing (echoing) user input to the browser without checking it first is something that should be avoided in secure forms processors: if a user entered the Javascript code <code>&lt;script&gt;alert('Error - Virus installed')&lt;/script&gt;</code> into the user name-field, the browser would execute the script on the form_handler.php page, just as if it had been coded by the developer; malicious code could be executed this way.  <code>filter_input()</code> was introduced in PHP 5.2. Users of earlier PHP versions could use the <code>htmlspecialchars()</code> function, or <a href="/wiki/Regular_expressions" class="mw-redirect" title="Regular expressions">regular expressions</a> to sanitize the user input before doing anything with it.
</p>
<h4><span class="mw-headline" id="Perl_programming_language">Perl programming language</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Form_(HTML)&amp;action=edit&amp;section=9" title="Edit section: Perl programming language">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p><a href="/wiki/Perl" title="Perl">Perl</a> is another language often used for <a href="/wiki/Web_development" title="Web development">web development</a>. Perl scripts are traditionally used as <a href="/wiki/Common_Gateway_Interface" title="Common Gateway Interface">Common Gateway Interface</a> applications (CGIs). In fact, Perl is such a common way to write CGIs that the two are often confused. CGIs may be written in other languages than Perl (compatibility with multiple languages is a design goal of the CGI protocol) and there are other ways to make Perl scripts interoperate with a <a href="/wiki/Web_server" title="Web server">web server</a> than using CGI (such as <a href="/wiki/FastCGI" title="FastCGI">FastCGI</a>, <a href="/wiki/Plack_(software)" title="Plack (software)">Plack</a> or <a href="/wiki/Apache_HTTP_Server" title="Apache HTTP Server">Apache</a>'s <a href="/wiki/Mod_perl" title="Mod perl">mod_perl</a>).
</p><p>Perl CGIs were once a very common way to write <a href="/wiki/Web_application" title="Web application">web applications</a>. However, many web hosts today effectively only support PHP, and developers of web applications often seek compatibility with them.
</p><p>A modern Perl 5 CGI using the CGI module with a form similar to the one above might look like:
</p><p><b>form_handler.pl</b>
</p>
<div class="mw-highlight mw-content-ltr" dir="ltr"><pre><span></span><span class="ch">#!/usr/bin/perl</span>
<span class="k">use</span> <span class="nn">strict</span><span class="p">;</span>
<span class="k">use</span> <span class="nn">CGI</span> <span class="sx">qw(:standard)</span><span class="p">;</span>

<span class="k">my</span> <span class="nv">$name</span><span class="o">=</span> <span class="n">param</span><span class="p">(</span><span class="s">&#39;firstname&#39;</span><span class="p">);</span>
<span class="k">print</span> <span class="n">header</span><span class="p">;</span>
<span class="k">print</span> <span class="n">html</span><span class="p">(</span>
  <span class="n">body</span><span class="p">(</span>
    <span class="n">p</span><span class="p">(</span><span class="s">&quot;Hello, $name!&quot;</span><span class="p">),</span>
  <span class="p">),</span>
<span class="p">);</span>
</pre></div>
<h4><span class="mw-headline" id="Form-to-email_scripts">Form-to-email scripts</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Form_(HTML)&amp;action=edit&amp;section=10" title="Edit section: Form-to-email scripts">edit</a><span class="mw-editsection-bracket">]</span></span></h4>
<p>Among the simplest and most commonly needed types of server-side script is that which simply emails the contents of a submitted form. This kind of script is frequently exploited by <a href="/wiki/Spammer" class="mw-redirect" title="Spammer">spammers</a>, however, and many of the most popular form-to-email scripts in use are vulnerable to hijacking for the purpose of sending spam emails. One of the most popular scripts of this type was <a rel="nofollow" class="external text" href="http://www.scriptarchive.com/formmail.html">"FormMail.pl"</a> made by <a href="/wiki/Matt%27s_Script_Archive" title="Matt&#39;s Script Archive">Matt's Script Archive</a>. Today, this script is no longer widely used in new development due to lack of updates, security concerns, and difficulty of configuration.
</p>
<h3><span class="mw-headline" id="Form_builders">Form builders</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Form_(HTML)&amp;action=edit&amp;section=11" title="Edit section: Form builders">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Some companies offer forms as a <a href="/wiki/Software_as_a_service" title="Software as a service">hosted service</a>. Usually, these companies give some kind of visual editor, reporting tools and infrastructure to create and host the forms, that can be embedded into webpages.<sup id="cite_ref-8" class="reference"><a href="#cite_note-8">&#91;8&#93;</a></sup> <a href="/wiki/Web_hosting" class="mw-redirect" title="Web hosting">Web hosting</a> companies provide templates to their clients as an add-on service.  Other form hosting services offer free contact forms that a user can install on their own website by pasting the service's code into the site's HTML.
</p>
