<p><b>Web server</b> refers to <a href="/wiki/Server_software" class="mw-redirect" title="Server software">server software</a>, or hardware dedicated to running said software, that can serve contents to the <a href="/wiki/World_Wide_Web" title="World Wide Web">World Wide Web</a>. A web server processes incoming network requests over <a href="/wiki/Hypertext_Transfer_Protocol" title="Hypertext Transfer Protocol">HTTP</a> and several other related protocols.<sup id="cite_ref-new_1-0" class="reference"><a href="#cite_note-new-1">&#91;1&#93;</a></sup>
</p>

<h2><span class="mw-headline" id="Overview">Overview</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_server&amp;action=edit&amp;section=1" title="Edit section: Overview">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The primary function of a web server is to store, process and deliver <a href="/wiki/Web_page" title="Web page">web pages</a> to <a href="/wiki/Client_(computing)" title="Client (computing)">clients</a>. The communication between client and server takes place using the <a href="/wiki/Hypertext_Transfer_Protocol" title="Hypertext Transfer Protocol">Hypertext Transfer Protocol (HTTP)</a>. Pages delivered are most frequently <a href="/wiki/HTML" title="HTML">HTML documents</a>, which may include <a href="/wiki/Image" title="Image">images</a>, <a href="/wiki/Style_sheet_(web_development)" title="Style sheet (web development)">style sheets</a> and <a href="/wiki/JavaScript" title="JavaScript">scripts</a> in addition to the text content. 
</p>
<div class="thumb tright"><div class="thumbinner" style="width:277px;"><a href="/wiki/File:Wikimedia_Foundation_Servers-8055_35.jpg" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/6/69/Wikimedia_Foundation_Servers-8055_35.jpg/275px-Wikimedia_Foundation_Servers-8055_35.jpg" width="275" height="183" class="thumbimage" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/6/69/Wikimedia_Foundation_Servers-8055_35.jpg/413px-Wikimedia_Foundation_Servers-8055_35.jpg 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/6/69/Wikimedia_Foundation_Servers-8055_35.jpg/550px-Wikimedia_Foundation_Servers-8055_35.jpg 2x" data-file-width="5184" data-file-height="3456" /></a>  <div class="thumbcaption"><div class="magnify"><a href="/wiki/File:Wikimedia_Foundation_Servers-8055_35.jpg" class="internal" title="Enlarge"></a></div>Multiple web servers may be used for a high traffic website; here, <a href="/wiki/Dell" title="Dell">Dell</a> servers are installed together being used for the <a href="/wiki/Wikimedia_Foundation" title="Wikimedia Foundation">Wikimedia Foundation</a>.</div></div></div>
<p>A <a href="/wiki/User_agent" title="User agent">user agent</a>, commonly a <a href="/wiki/Web_browser" title="Web browser">web browser</a> or <a href="/wiki/Web_crawler" title="Web crawler">web crawler</a>, initiates communication by making a <a href="/wiki/Hypertext_Transfer_Protocol#Request_message" title="Hypertext Transfer Protocol">request</a> for a specific resource using HTTP and the server responds with the content of that resource or an <a href="/wiki/List_of_HTTP_status_codes#4xx_Client_Error" title="List of HTTP status codes">error message</a> if unable to do so. The resource is typically a real file on the server's <a href="/wiki/Secondary_memory" class="mw-redirect" title="Secondary memory">secondary storage</a>, but this is not necessarily the case and depends on how the web server is <a href="/wiki/Implementation" title="Implementation">implemented</a>.
</p><p>While the primary function is to serve content, a full implementation of HTTP also includes ways of receiving content from clients. This feature is used for submitting <a href="/wiki/Form_(web)" class="mw-redirect" title="Form (web)">web forms</a>, including <a href="/wiki/Upload" title="Upload">uploading</a> of files.
</p><p>Many generic web servers also support <a href="/wiki/Server-side_scripting" title="Server-side scripting">server-side scripting</a> using <a href="/wiki/Active_Server_Pages" title="Active Server Pages">Active Server Pages</a> (ASP), <a href="/wiki/PHP" title="PHP">PHP</a>, or other <a href="/wiki/Scripting_language" title="Scripting language">scripting languages</a>. This means that the behaviour of the web server can be scripted in separate files, while the actual server software remains unchanged. Usually, this function is used to generate HTML documents <a href="/wiki/Dynamic_web_page" title="Dynamic web page">dynamically</a> ("on-the-fly") as opposed to returning <a href="/wiki/Static_web_page" title="Static web page">static documents</a>. The former is primarily used for retrieving or modifying information from <a href="/wiki/Database" title="Database">databases</a>. The latter is typically much faster and more easily <a href="/wiki/Web_cache" title="Web cache">cached</a> but cannot deliver <a href="/wiki/Dynamic_content" class="mw-redirect" title="Dynamic content">dynamic content</a>.
</p><p>Web servers are not only used for serving the <a href="/wiki/World_Wide_Web" title="World Wide Web">World Wide Web</a>. They can also be found <a href="/wiki/Embedded_system" title="Embedded system">embedded</a> in devices such as <a href="/wiki/Printer_(computing)" title="Printer (computing)">printers</a>, <a href="/wiki/Router_(computing)" title="Router (computing)">routers</a>, <a href="/wiki/Webcam" title="Webcam">webcams</a> and serving only a <a href="/wiki/Local_area_network" title="Local area network">local network</a>. The web server may then be used as a part of a system for monitoring or administering the device in question. This usually means that no additional software has to be installed on the client computer since only a web browser is required (which now is included with most <a href="/wiki/Operating_system" title="Operating system">operating systems</a>).
</p>
<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_server&amp;action=edit&amp;section=2" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="thumb tright"><div class="thumbinner" style="width:222px;"><a href="/wiki/File:First_Web_Server.jpg" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/d/d1/First_Web_Server.jpg/220px-First_Web_Server.jpg" width="220" height="165" class="thumbimage" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/d/d1/First_Web_Server.jpg/330px-First_Web_Server.jpg 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/d/d1/First_Web_Server.jpg/440px-First_Web_Server.jpg 2x" data-file-width="2048" data-file-height="1536" /></a>  <div class="thumbcaption"><div class="magnify"><a href="/wiki/File:First_Web_Server.jpg" class="internal" title="Enlarge"></a></div>The world's first web server, a <a href="/wiki/NeXT_Computer" title="NeXT Computer">NeXT Computer</a> workstation with Ethernet, 1990. The case label reads: "This machine is a server. DO NOT POWER IT DOWN!!"</div></div></div>
<div class="thumb tright"><div class="thumbinner" style="width:222px;"><a href="/wiki/File:Cobalt_Qube_3_Front.jpg" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/6/6d/Cobalt_Qube_3_Front.jpg/220px-Cobalt_Qube_3_Front.jpg" width="220" height="165" class="thumbimage" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/6/6d/Cobalt_Qube_3_Front.jpg/330px-Cobalt_Qube_3_Front.jpg 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/6/6d/Cobalt_Qube_3_Front.jpg/440px-Cobalt_Qube_3_Front.jpg 2x" data-file-width="1600" data-file-height="1200" /></a>  <div class="thumbcaption"><div class="magnify"><a href="/wiki/File:Cobalt_Qube_3_Front.jpg" class="internal" title="Enlarge"></a></div>Sun's <a href="/wiki/Cobalt_Qube" title="Cobalt Qube">Cobalt Qube</a> 3 - a computer <a href="/wiki/Server_appliance" class="mw-redirect" title="Server appliance">server appliance</a> (2002, discontinued)</div></div></div>
<p>In 1989 <a href="/wiki/Sir_Tim_Berners-Lee" class="mw-redirect" title="Sir Tim Berners-Lee">Sir Tim Berners-Lee</a> proposed a new project to his employer <a href="/wiki/CERN" title="CERN">CERN</a>, with the goal of easing the exchange of information between scientists by using a <a href="/wiki/Hypertext" title="Hypertext">hypertext</a> system. The project resulted in Berners-Lee writing two programs in 1990:
</p>
<ul><li>A <a href="/wiki/Web_browser" title="Web browser">browser</a> called <a href="/wiki/WorldWideWeb" title="WorldWideWeb">WorldWideWeb</a></li>
<li>The world's first web server, later known as <a href="/wiki/CERN_httpd" title="CERN httpd">CERN httpd</a>, which ran on <a href="/wiki/NeXTSTEP" title="NeXTSTEP">NeXTSTEP</a></li></ul>
<p>Between 1991 and 1994, the simplicity and effectiveness of early technologies used to surf and exchange data through the World Wide Web helped to port them to many different operating systems and spread their use among scientific organizations and universities, and subsequently to the industry.
</p><p>In 1994 Berners-Lee decided to constitute the <a href="/wiki/World_Wide_Web_Consortium" title="World Wide Web Consortium">World Wide Web Consortium</a> (W3C) to regulate the further development of the many technologies involved (<a href="/wiki/HTTP" class="mw-redirect" title="HTTP">HTTP</a>, <a href="/wiki/HTML" title="HTML">HTML</a>, etc.) through a standardization process.
</p>
<h2><span class="mw-headline" id="Path_translation">Path translation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_server&amp;action=edit&amp;section=3" title="Edit section: Path translation">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Web servers are able to map the path component of a <a href="/wiki/Uniform_Resource_Locator" class="mw-redirect" title="Uniform Resource Locator">Uniform Resource Locator</a> (<b>URL</b>) into:
</p>
<ul><li>A local <a href="/wiki/File_system" title="File system">file system</a> resource (for static requests)</li>
<li>An internal or external program name (for dynamic requests)</li></ul>
<p>For a <i>static request</i> the URL path specified by the client is relative to the web server's root directory.
</p><p>Consider the following URL as it would be requested by a client:
</p>
<pre>http://www.example.com/path/file.html
</pre>
<p>The client's <a href="/wiki/User_agent" title="User agent">user agent</a> will translate it into a connection to <tt>www.example.com</tt> with the following HTTP 1.1 request:
</p>
<pre>GET /path/file.html HTTP/1.1
Host: www.example.com
</pre>
<p>The web server on <tt>www.example.com</tt> will append the given path to the path of its root directory. On an <a href="/wiki/Apache_server" class="mw-redirect" title="Apache server">Apache server</a>, this is commonly <tt>/home/www</tt> (on <a href="/wiki/Unix" title="Unix">Unix</a> machines, usually <tt>/var/www</tt>). The result is the local file system resource:
</p>
<pre>/home/www<b>/path/file.html</b>
</pre>
<p>The web server then reads the <a href="/wiki/Computer_file" title="Computer file">file</a>, if it exists, and sends a response to the client's web browser. The response will describe the content of the file and contain the file itself or an error message will return saying that the file does not exist or is unavailable.
</p>
<h2><span class="mw-headline" id="Kernel-mode_and_user-mode_web_servers">Kernel-mode and user-mode web servers</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_server&amp;action=edit&amp;section=4" title="Edit section: Kernel-mode and user-mode web servers">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>A web server can be either incorporated into the <a href="/wiki/Operating_system" title="Operating system">OS</a> <a href="/wiki/Kernel_(computing)" class="mw-redirect" title="Kernel (computing)">kernel</a>, or in <a href="/wiki/User_space" title="User space">user space</a> (like other regular applications).
</p><p>Web servers that run in <a href="/wiki/User-mode" class="mw-redirect" title="User-mode">user-mode</a> have to ask the system for permission to use more memory or more CPU resources. Not only do these requests to the kernel take time, but they are not always satisfied because the system reserves resources for its own usage and has the responsibility to share hardware resources with all the other running applications. Executing in user mode can also mean useless buffer copies which are another handicap for user-mode web servers.
</p>
<h2><span class="mw-headline" id="Load_limits">Load limits</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_server&amp;action=edit&amp;section=5" title="Edit section: Load limits">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>A web server (program) has defined load limits, because it can handle only a limited number of concurrent client connections (usually between 2 and 80,000, by default between 500 and 1,000) per <a href="/wiki/IP_address" title="IP address">IP address</a> (and TCP port) and it can serve only a certain maximum number of <i><span id="requests_per_second">requests per second</span></i> (RPS, also known as <a href="/wiki/Queries_per_second" title="Queries per second">queries per second</a> or QPS) depending on:
</p>
<ul><li>its own settings,</li>
<li>the HTTP request type,</li>
<li>whether the content is static or dynamic,</li>
<li>whether the content is <a href="/wiki/Web_cache" title="Web cache">cached</a>, and</li>
<li>the <a href="/wiki/Personal_computer_hardware" class="mw-redirect" title="Personal computer hardware">hardware</a> and <a href="/wiki/Computer_software" class="mw-redirect" title="Computer software">software</a> limitations of the OS of the computer on which the web server runs.</li></ul>
<p>When a web server is near to or over its limit, it becomes unresponsive.
</p>
<h3><span class="mw-headline" id="Causes_of_overload">Causes of overload</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_server&amp;action=edit&amp;section=6" title="Edit section: Causes of overload">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>At any time web servers can be overloaded due to:
</p>
<ul><li>Excess legitimate web traffic. Thousands or even millions of clients connecting to the web site in a short interval, e.g., <a href="/wiki/Slashdot_effect" title="Slashdot effect">Slashdot effect</a>;</li>
<li><a href="/wiki/Distributed_Denial_of_Service" class="mw-redirect" title="Distributed Denial of Service">Distributed Denial of Service</a> attacks. A denial-of-service attack (DoS attack) or distributed denial-of-service attack (DDoS attack) is an attempt to make a computer or network resource unavailable to its intended users;</li>
<li><a href="/wiki/Computer_worm" title="Computer worm">Computer worms</a> that sometimes cause abnormal traffic because of millions of infected computers (not coordinated among them)</li>
<li><a href="/wiki/XSS_Worm" class="mw-redirect" title="XSS Worm">XSS viruses</a> can cause high traffic because of millions of infected browsers or web servers;</li>
<li><a href="/wiki/Internet_bot" title="Internet bot">Internet bots</a> Traffic not filtered/limited on large web sites with very few resources (bandwidth, etc.);</li>
<li><a href="/wiki/Internet" title="Internet">Internet</a> (network) slowdowns, so that client requests are served more slowly and the number of connections increases so much that server limits are reached;</li>
<li>Web servers (<a href="/wiki/Computer" title="Computer">computers</a>) partial unavailability. This can happen because of required or urgent maintenance or upgrade, hardware or software failures, <a href="/wiki/Front_and_back_ends" title="Front and back ends">back-end</a> (e.g., <a href="/wiki/Database" title="Database">database</a>) failures, etc.; in these cases the remaining web servers get too much traffic and become overloaded.</li></ul>
<h3><span class="mw-headline" id="Symptoms_of_overload">Symptoms of overload</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_server&amp;action=edit&amp;section=7" title="Edit section: Symptoms of overload">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The symptoms of an overloaded web server are:
</p>
<ul><li>Requests are served with (possibly long) delays (from 1 second to a few hundred seconds).</li>
<li>The web server returns an <a href="/wiki/List_of_HTTP_status_codes" title="List of HTTP status codes">HTTP error code</a>, such as 500, 502, 503, 504, 408, or even <a href="/wiki/HTTP_404" title="HTTP 404">404</a>, which is inappropriate for an overload condition.</li>
<li>The web server refuses or resets (interrupts) <a href="/wiki/Transmission_control_protocol" class="mw-redirect" title="Transmission control protocol">TCP</a> connections before it returns any content.</li>
<li>In very rare cases, the web server returns only a part of the requested content. This behavior can be considered a <a href="/wiki/Software_bug" title="Software bug">bug</a>, even if it usually arises as a symptom of overload.</li></ul>
<h3><span class="mw-headline" id="Anti-overload_techniques">Anti-overload techniques</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_server&amp;action=edit&amp;section=8" title="Edit section: Anti-overload techniques">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>To partially overcome above average load limits and to prevent overload, most popular web sites use common techniques like:
</p>
<ul><li>Managing network traffic, by using:
<ul><li><a href="/wiki/Firewall_(computing)" title="Firewall (computing)">Firewalls</a> to block unwanted traffic coming from bad IP sources or having bad patterns</li>
<li>HTTP traffic managers to drop, redirect or rewrite requests having bad <a href="/wiki/HTTP" class="mw-redirect" title="HTTP">HTTP</a> patterns</li>
<li><a href="/wiki/Bandwidth_management" title="Bandwidth management">Bandwidth management</a> and <a href="/wiki/Traffic_shaping" title="Traffic shaping">traffic shaping</a>, in order to smooth down peaks in network usage</li></ul></li>
<li>Deploying <a href="/wiki/Web_cache" title="Web cache">web cache</a> techniques</li>
<li>Using different <a href="/wiki/Domain_name" title="Domain name">domain names</a> to serve different (static and dynamic) content by separate web servers, e.g.:
<ul><li>http://images.example.com</li>
<li>http://www.example.com</li></ul></li>
<li>Using different domain names or computers to separate big files from small and medium-sized files; the idea is to be able to fully <a href="/wiki/Web_cache" title="Web cache">cache</a> small and medium-sized files and to efficiently serve big or huge (over 10 - 1000 MB) files by using different settings</li>
<li>Using many internet servers (programs) per computer, each one bound to its own <a href="/wiki/Network_card" class="mw-redirect" title="Network card">network card</a> and <a href="/wiki/IP_address" title="IP address">IP address</a></li>
<li>Using many internet servers (computers) that are grouped together behind a <a href="/wiki/Load_balancing_(computing)" title="Load balancing (computing)">load balancer</a> so that they act or are seen as one big web server</li>
<li>Adding more hardware resources (i.e. <a href="/wiki/RAM" class="mw-redirect" title="RAM">RAM</a>, <a href="/wiki/Disk_storage" title="Disk storage">disks</a>) to each computer</li>
<li>Tuning OS parameters for hardware capabilities and usage</li>
<li>Using more efficient <a href="/wiki/Computer_program" title="Computer program">computer programs</a> for web servers, etc.</li>
<li>Using other <a href="/wiki/Workaround" title="Workaround">workarounds</a>, especially if dynamic content is involved</li></ul>
