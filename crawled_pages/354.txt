<p>The <b>File Transfer Protocol</b> (<b>FTP</b>) is a standard <a href="/wiki/Network_protocol" class="mw-redirect" title="Network protocol">network protocol</a> used for the transfer of <a href="/wiki/Computer_file" title="Computer file">computer files</a> between <a href="/wiki/Client%E2%80%93server_model" title="Client–server model">a client and server</a> on a <a href="/wiki/Computer_network" title="Computer network">computer network</a>.
</p><p>FTP is built on a client-server model architecture using separate control and data connections between the client and the server.<sup id="cite_ref-for_1-0" class="reference"><a href="#cite_note-for-1">&#91;1&#93;</a></sup> FTP users may authenticate themselves with a <a href="/wiki/Clear_text" class="mw-redirect" title="Clear text">clear-text</a> sign-in protocol, normally in the form of a username and password, but can connect anonymously if the server is configured to allow it. For secure transmission that protects the username and password, and encrypts the content, FTP is often <a href="/wiki/File_Transfer_Protocol#Security" title="File Transfer Protocol">secured</a> with <a href="/wiki/Transport_Layer_Security" title="Transport Layer Security">SSL/TLS</a> (<a href="/wiki/FTPS" title="FTPS">FTPS</a>) or replaced with <a href="/wiki/SSH_File_Transfer_Protocol" title="SSH File Transfer Protocol">SSH File Transfer Protocol</a> (SFTP).
</p><p>The first FTP client applications were <a href="/wiki/Command-line_interface" title="Command-line interface">command-line programs</a> developed before <a href="/wiki/Operating_system" title="Operating system">operating systems</a> had <a href="/wiki/Graphical_user_interface" title="Graphical user interface">graphical user interfaces</a>, and are still shipped with most <a href="/wiki/Windows" class="mw-redirect" title="Windows">Windows</a>, <a href="/wiki/Unix" title="Unix">Unix</a>, and <a href="/wiki/Linux" title="Linux">Linux</a> operating systems.<sup id="cite_ref-tcpip_2-0" class="reference"><a href="#cite_note-tcpip-2">&#91;2&#93;</a></sup><sup id="cite_ref-net+_3-0" class="reference"><a href="#cite_note-net+-3">&#91;3&#93;</a></sup> Many FTP clients and automation utilities have since been developed for <a href="/wiki/Desktop_computer" title="Desktop computer">desktops</a>, servers, mobile devices, and hardware, and FTP has been incorporated into productivity applications, such as <a href="/wiki/HTML_editor" title="HTML editor">web page editors</a>.
</p>

<h2><span class="mw-headline" id="History_of_FTP_servers">History of FTP servers</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=File_Transfer_Protocol&amp;action=edit&amp;section=1" title="Edit section: History of FTP servers">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The original specification for the File Transfer Protocol was written by <a href="/wiki/Abhay_Bhushan" title="Abhay Bhushan">Abhay Bhushan</a> and published as <a href="/wiki/Request_for_Comments_(identifier)" class="mw-redirect" title="Request for Comments (identifier)">RFC</a>&#160;<a rel="nofollow" class="external text" href="https://tools.ietf.org/html/rfc114">114</a> on 16 April 1971. Until 1980, FTP ran on <a href="/wiki/Network_Control_Program" title="Network Control Program">NCP</a>, the predecessor of <a href="/wiki/Internet_protocol_suite" title="Internet protocol suite">TCP/IP</a>.<sup id="cite_ref-tcpip_2-1" class="reference"><a href="#cite_note-tcpip-2">&#91;2&#93;</a></sup> The protocol was later replaced by a TCP/IP version, <a href="/wiki/Request_for_Comments_(identifier)" class="mw-redirect" title="Request for Comments (identifier)">RFC</a>&#160;<a rel="nofollow" class="external text" href="https://tools.ietf.org/html/rfc765">765</a> (June 1980) and <a href="/wiki/Request_for_Comments_(identifier)" class="mw-redirect" title="Request for Comments (identifier)">RFC</a>&#160;<a rel="nofollow" class="external text" href="https://tools.ietf.org/html/rfc959">959</a> (October 1985), the current specification. Several proposed standards amend <a href="/wiki/Request_for_Comments_(identifier)" class="mw-redirect" title="Request for Comments (identifier)">RFC</a>&#160;<a rel="nofollow" class="external text" href="https://tools.ietf.org/html/rfc959">959</a>, for example <a href="/wiki/Request_for_Comments_(identifier)" class="mw-redirect" title="Request for Comments (identifier)">RFC</a>&#160;<a rel="nofollow" class="external text" href="https://tools.ietf.org/html/rfc1579">1579</a> (February 1994) enables Firewall-Friendly FTP (passive mode), <a href="/wiki/Request_for_Comments_(identifier)" class="mw-redirect" title="Request for Comments (identifier)">RFC</a>&#160;<a rel="nofollow" class="external text" href="https://tools.ietf.org/html/rfc2228">2228</a> (June 1997) proposes security extensions, <a href="/wiki/Request_for_Comments_(identifier)" class="mw-redirect" title="Request for Comments (identifier)">RFC</a>&#160;<a rel="nofollow" class="external text" href="https://tools.ietf.org/html/rfc2428">2428</a> (September 1998) adds support for <a href="/wiki/IPv6" title="IPv6">IPv6</a> and defines a new type of passive mode.<sup id="cite_ref-clark_4-0" class="reference"><a href="#cite_note-clark-4">&#91;4&#93;</a></sup> FTP has two modes: active mode, data transfer, using TCP port 20 and control, command, mode using TCP port 21.<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">&#91;5&#93;</a></sup>
</p>
<h2><span class="mw-headline" id="Protocol_overview">Protocol overview</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=File_Transfer_Protocol&amp;action=edit&amp;section=2" title="Edit section: Protocol overview">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="Communication_and_data_transfer">Communication and data transfer</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=File_Transfer_Protocol&amp;action=edit&amp;section=3" title="Edit section: Communication and data transfer">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div class="thumb tright"><div class="thumbinner" style="width:222px;"><a href="/wiki/File:Passive_FTP_Verbindung.svg" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/8/84/Passive_FTP_Verbindung.svg/220px-Passive_FTP_Verbindung.svg.png" width="220" height="264" class="thumbimage" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/8/84/Passive_FTP_Verbindung.svg/330px-Passive_FTP_Verbindung.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/8/84/Passive_FTP_Verbindung.svg/440px-Passive_FTP_Verbindung.svg.png 2x" data-file-width="359" data-file-height="431" /></a>  <div class="thumbcaption"><div class="magnify"><a href="/wiki/File:Passive_FTP_Verbindung.svg" class="internal" title="Enlarge"></a></div>Illustration of starting a passive connection using port 21</div></div></div>
<p>FTP may run in <i>active</i> or <i>passive</i> mode, which determines how the data connection is established.<sup id="cite_ref-slack_6-0" class="reference"><a href="#cite_note-slack-6">&#91;6&#93;</a></sup> In both cases, the client creates a TCP control connection from a random, usually an unprivileged, <a href="/wiki/Port_(computer_networking)" title="Port (computer networking)">port</a> N to the FTP server command port 21.
</p>
<ul><li>In active mode, the client starts listening for incoming data connections from the server on port M. It sends the FTP command PORT M to inform the server on which port it is listening. The server then initiates a data channel to the client from its port 20, the FTP server data port.</li>
<li>In situations where the client is behind a <a href="/wiki/Firewall_(computing)" title="Firewall (computing)">firewall</a> and unable to accept incoming TCP connections, <i>passive mode</i> may be used. In this mode, the client uses the control connection to send a PASV command to the server and then receives a server IP address and server port number from the server,<sup id="cite_ref-slack_6-1" class="reference"><a href="#cite_note-slack-6">&#91;6&#93;</a></sup> which the client then uses to open a data connection from an arbitrary client port to the server IP address and server port number received.<sup id="cite_ref-postel_7-0" class="reference"><a href="#cite_note-postel-7">&#91;7&#93;</a></sup></li></ul>
<p>Both modes were updated in September 1998 to support <a href="/wiki/IPv6" title="IPv6">IPv6</a>. Further changes were introduced to the passive mode at that time, updating it to <i>extended passive mode</i>.<sup id="cite_ref-8" class="reference"><a href="#cite_note-8">&#91;8&#93;</a></sup>
</p><p>The server responds over the control connection with <a href="/wiki/List_of_FTP_server_return_codes" title="List of FTP server return codes">three-digit status codes</a> in ASCII with an optional text message. For example, "200" (or "200 OK") means that the last command was successful. The numbers represent the code for the response and the optional text represents a human-readable explanation or request (e.g. &lt;Need account for storing file&gt;).<sup id="cite_ref-for_1-1" class="reference"><a href="#cite_note-for-1">&#91;1&#93;</a></sup> An ongoing transfer of file data over the data connection can be aborted using an interrupt message sent over the control connection.
</p><p>While transferring data over the network, four data representations can be used:<sup id="cite_ref-tcpip_2-2" class="reference"><a href="#cite_note-tcpip-2">&#91;2&#93;</a></sup><sup id="cite_ref-net+_3-1" class="reference"><a href="#cite_note-net+-3">&#91;3&#93;</a></sup><sup id="cite_ref-clark_4-1" class="reference"><a href="#cite_note-clark-4">&#91;4&#93;</a></sup>
</p>
<ul><li><a href="/wiki/ASCII" title="ASCII">ASCII</a> mode: Used for text. Data is converted, if needed, from the sending host's character representation to <a href="/wiki/Extended_ASCII" title="Extended ASCII">"8-bit ASCII"</a> before transmission, and (again, if necessary) to the receiving host's character representation. As a consequence, this mode is inappropriate for files that contain data other than plain text.</li>
<li>Image mode (commonly called <a href="/wiki/Binary_data" title="Binary data">Binary</a> mode): The sending machine sends each file <a href="/wiki/Byte" title="Byte">byte</a> by byte, and the recipient stores the <a href="/wiki/Bytestream" class="mw-redirect" title="Bytestream">bytestream</a> as it receives it. (Image mode support has been recommended for all implementations of FTP).</li>
<li><a href="/wiki/EBCDIC" title="EBCDIC">EBCDIC</a> mode: Used for plain text between hosts using the EBCDIC character set.</li>
<li>Local mode: Allows two computers with identical setups to send data in a proprietary format without the need to convert it to ASCII.</li></ul>
<p>For text files, different format control and record structure options are provided. These features were designed to facilitate files containing <a href="/wiki/Telnet" title="Telnet">Telnet</a> or <a href="/wiki/ASA_carriage_control_characters" title="ASA carriage control characters">ASA</a>.
</p><p>Data transfer can be done in any of three modes:<sup id="cite_ref-for_1-2" class="reference"><a href="#cite_note-for-1">&#91;1&#93;</a></sup><sup id="cite_ref-tcpip_2-3" class="reference"><a href="#cite_note-tcpip-2">&#91;2&#93;</a></sup>
</p>
<ul><li>Stream mode: Data is sent as a continuous stream, relieving FTP from doing any processing. Rather, all processing is left up to <a href="/wiki/Transmission_Control_Protocol" title="Transmission Control Protocol">TCP</a>. No End-of-file indicator is needed, unless the data is divided into <a href="/wiki/Record_(computer_science)" title="Record (computer science)">records</a>.</li>
<li>Block mode: FTP breaks the data into several blocks (block header, byte count, and data field) and then passes it on to TCP.<sup id="cite_ref-clark_4-2" class="reference"><a href="#cite_note-clark-4">&#91;4&#93;</a></sup></li>
<li>Compressed mode: Data is compressed using a simple algorithm (usually <a href="/wiki/Run-length_encoding" title="Run-length encoding">run-length encoding</a>).</li></ul>
<p>Some FTP software also implements a <a href="/wiki/DEFLATE" title="DEFLATE">DEFLATE</a>-based compressed mode, sometimes called "Mode Z" after the command that enables it. This mode was described in an <a href="/wiki/Internet_Draft" title="Internet Draft">Internet Draft</a>, but not standardized.<sup id="cite_ref-9" class="reference"><a href="#cite_note-9">&#91;9&#93;</a></sup>
</p>
<h3><span class="mw-headline" id="Login">Login</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=File_Transfer_Protocol&amp;action=edit&amp;section=4" title="Edit section: Login">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>FTP login uses normal username and password scheme for granting access.<sup id="cite_ref-tcpip_2-4" class="reference"><a href="#cite_note-tcpip-2">&#91;2&#93;</a></sup> The username is sent to the server using the USER command, and the password is sent using the PASS command.<sup id="cite_ref-tcpip_2-5" class="reference"><a href="#cite_note-tcpip-2">&#91;2&#93;</a></sup> This sequence is unencrypted "on the wire", so may be vulnerable to a network <a href="/wiki/Sniffing_attack" title="Sniffing attack">sniffing attack</a>.<sup id="cite_ref-10" class="reference"><a href="#cite_note-10">&#91;10&#93;</a></sup> If the information provided by the client is accepted by the server, the server will send a greeting to the client and the session will commence.<sup id="cite_ref-tcpip_2-6" class="reference"><a href="#cite_note-tcpip-2">&#91;2&#93;</a></sup> If the server supports it, users may log in without providing login credentials, but the same server may authorize only limited access for such sessions.<sup id="cite_ref-tcpip_2-7" class="reference"><a href="#cite_note-tcpip-2">&#91;2&#93;</a></sup>
</p>
<h3><span class="mw-headline" id="Anonymous_FTP">Anonymous FTP</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=File_Transfer_Protocol&amp;action=edit&amp;section=5" title="Edit section: Anonymous FTP">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>A host that provides an FTP service may provide <a href="/wiki/Anonymity" title="Anonymity">anonymous</a> FTP access.<sup id="cite_ref-tcpip_2-8" class="reference"><a href="#cite_note-tcpip-2">&#91;2&#93;</a></sup> Users typically log into the service with an 'anonymous' (lower-case and case-sensitive in some FTP servers) account when prompted for user name. Although users are commonly asked to send their <a href="/wiki/Email" title="Email">email</a> address instead of a password,<sup id="cite_ref-net+_3-2" class="reference"><a href="#cite_note-net+-3">&#91;3&#93;</a></sup> no verification is actually performed on the supplied data.<sup id="cite_ref-rfc16_11-0" class="reference"><a href="#cite_note-rfc16-11">&#91;11&#93;</a></sup> Many FTP hosts whose purpose is to provide software updates will allow anonymous logins.<sup id="cite_ref-net+_3-3" class="reference"><a href="#cite_note-net+-3">&#91;3&#93;</a></sup>
</p>
<h3><span class="mw-headline" id="NAT_and_firewall_traversal">NAT and firewall traversal</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=File_Transfer_Protocol&amp;action=edit&amp;section=6" title="Edit section: NAT and firewall traversal">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>FTP normally transfers data by having the server connect back to the client, after the PORT command is sent by the client. This is problematic for both <a href="/wiki/Network_address_translation" title="Network address translation">NATs</a> and firewalls, which do not allow connections from the Internet towards internal hosts.<sup id="cite_ref-ncftp_12-0" class="reference"><a href="#cite_note-ncftp-12">&#91;12&#93;</a></sup> For NATs, an additional complication is that the representation of the IP addresses and port number in the PORT command refer to the internal host's IP address and port, rather than the public IP address and port of the NAT.
</p><p>There are two approaches to solve this problem. One is that the FTP client and FTP server use the PASV command, which causes the data connection to be established from the FTP client to the server.<sup id="cite_ref-ncftp_12-1" class="reference"><a href="#cite_note-ncftp-12">&#91;12&#93;</a></sup> This is widely used by modern FTP clients. Another approach is for the NAT to alter the values of the PORT command, using an <a href="/wiki/Application-level_gateway" title="Application-level gateway">application-level gateway</a> for this purpose.<sup id="cite_ref-ncftp_12-2" class="reference"><a href="#cite_note-ncftp-12">&#91;12&#93;</a></sup>
</p>
<h3><span class="mw-headline" id="Differences_from_HTTP">Differences from HTTP</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=File_Transfer_Protocol&amp;action=edit&amp;section=7" title="Edit section: Differences from HTTP">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p><a href="/wiki/Hypertext_Transfer_Protocol" title="Hypertext Transfer Protocol">HTTP</a> essentially fixes the bugs in FTP that made it inconvenient to use for many small ephemeral transfers as are typical in web pages.
</p><p>FTP has a stateful control connection which maintains a current working directory and other flags, and each transfer requires a secondary connection through which the data are transferred. In "passive" mode this secondary connection is from client to server, whereas in the default "active" mode this connection is from server to client. This apparent role reversal when in active mode, and random port numbers for all transfers, is why firewalls and NAT gateways have such a hard time with FTP. HTTP is stateless and multiplexes control and data over a single connection from client to server on well-known port numbers, which trivially passes through NAT gateways and is simple for firewalls to manage.
</p><p>Setting up an FTP control connection is quite slow due to the round-trip delays of sending all of the required commands and awaiting responses, so it is customary to bring up a control connection and hold it open for multiple file transfers rather than drop and re-establish the session afresh each time. In contrast, HTTP originally dropped the connection after each transfer because doing so was so cheap. While HTTP has subsequently gained the ability to reuse the TCP connection for multiple transfers, the conceptual model is still of independent requests rather than a session.
</p><p>When FTP is transferring over the data connection, the control connection is idle. If the transfer takes too long, the firewall or NAT may decide that the control connection is dead and stop tracking it, effectively breaking the connection and confusing the download. The single HTTP connection is only idle  between requests and it is normal and expected for such connections to be dropped after a time-out.
</p>
<h2><span class="mw-headline" id="Web_browser_support">Web browser support</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=File_Transfer_Protocol&amp;action=edit&amp;section=8" title="Edit section: Web browser support">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Most common <a href="/wiki/Web_browser" title="Web browser">web browsers</a> can retrieve files hosted on FTP servers, although they may not support protocol extensions such as <a href="/wiki/FTPS" title="FTPS">FTPS</a>.<sup id="cite_ref-net+_3-4" class="reference"><a href="#cite_note-net+-3">&#91;3&#93;</a></sup><sup id="cite_ref-matthews_13-0" class="reference"><a href="#cite_note-matthews-13">&#91;13&#93;</a></sup> When an FTP—rather than an HTTP—<a href="/wiki/URL" title="URL">URL</a> is supplied, the accessible contents on the remote server are presented in a manner that is similar to that used for other web content. A full-featured FTP client can be run within <a href="/wiki/Firefox" title="Firefox">Firefox</a> in the form of an extension called <a href="/wiki/FireFTP" title="FireFTP">FireFTP</a>.
</p>
<h3><span class="mw-headline" id="Syntax">Syntax</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=File_Transfer_Protocol&amp;action=edit&amp;section=9" title="Edit section: Syntax">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>FTP URL syntax is described in <a href="/wiki/Request_for_Comments_(identifier)" class="mw-redirect" title="Request for Comments (identifier)">RFC</a>&#160;<a rel="nofollow" class="external text" href="https://tools.ietf.org/html/rfc1738">1738</a>, taking the form: <code>ftp://[user[:password]@]host[:port]/url-path</code> (the bracketed parts are optional).
</p><p>For example, the URL ftp://public.ftp-servers.example.com/mydirectory/myfile.txt represents the file <i>myfile.txt</i> from the directory <i>mydirectory</i> on the server <i>public.ftp-servers.example.com</i> as an FTP resource. The URL ftp://user001:secretpassword@private.ftp-servers.example.com/mydirectory/myfile.txt adds a specification of the username and password that must be used to access this resource.
</p><p>More details on specifying a username and password may be found in the browsers' documentation (e.g., <a href="/wiki/Firefox" title="Firefox">Firefox</a><sup id="cite_ref-14" class="reference"><a href="#cite_note-14">&#91;14&#93;</a></sup> and <a href="/wiki/Internet_Explorer" title="Internet Explorer">Internet Explorer</a><sup id="cite_ref-15" class="reference"><a href="#cite_note-15">&#91;15&#93;</a></sup>). By default, most web browsers use passive (PASV) mode, which more easily traverses end-user firewalls.
</p><p>Some variation has existed in how different browsers treat path resolution in cases where there is a non-root home directory for a user.<sup id="cite_ref-16" class="reference"><a href="#cite_note-16">&#91;16&#93;</a></sup>
</p>
<h2><span class="mw-headline" id="Security">Security</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=File_Transfer_Protocol&amp;action=edit&amp;section=10" title="Edit section: Security">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>FTP was not designed to be a secure protocol, and has many security weaknesses.<sup id="cite_ref-nurdle_17-0" class="reference"><a href="#cite_note-nurdle-17">&#91;17&#93;</a></sup> In May 1999, the authors of <a href="/wiki/Request_for_Comments_(identifier)" class="mw-redirect" title="Request for Comments (identifier)">RFC</a>&#160;<a rel="nofollow" class="external text" href="https://tools.ietf.org/html/rfc2577">2577</a> listed a vulnerability to the following problems:
</p>
<ul><li><a href="/wiki/Brute_force_attack" class="mw-redirect" title="Brute force attack">Brute force attack</a></li>
<li><a href="/wiki/FTP_bounce_attack" title="FTP bounce attack">FTP bounce attack</a></li>
<li><a href="/wiki/Packet_capture" class="mw-redirect" title="Packet capture">Packet capture</a></li>
<li>Port stealing (guessing the next open port and usurping a legitimate connection)</li>
<li><a href="/wiki/Spoofing_attack" title="Spoofing attack">Spoofing attack</a></li>
<li><a href="/wiki/Username" class="mw-redirect" title="Username">Username</a> enumeration</li></ul>
<p>FTP does not encrypt its traffic; all transmissions are in clear text, and usernames, passwords, commands and data can be read by anyone able to perform packet capture (<a href="/wiki/Packet_analyzer" title="Packet analyzer">sniffing</a>) on the network.<sup id="cite_ref-tcpip_2-9" class="reference"><a href="#cite_note-tcpip-2">&#91;2&#93;</a></sup><sup id="cite_ref-nurdle_17-1" class="reference"><a href="#cite_note-nurdle-17">&#91;17&#93;</a></sup> This problem is common to many of the Internet Protocol specifications (such as <a href="/wiki/SMTP" class="mw-redirect" title="SMTP">SMTP</a>, <a href="/wiki/Telnet" title="Telnet">Telnet</a>, POP and IMAP) that were designed prior to the creation of encryption mechanisms such as <a href="/wiki/Transport_Layer_Security" title="Transport Layer Security">TLS</a> or SSL.<sup id="cite_ref-clark_4-3" class="reference"><a href="#cite_note-clark-4">&#91;4&#93;</a></sup>
</p><p>Common solutions to this problem include:
</p>
<ol><li>Using the secure versions of the insecure protocols, e.g., <a href="/wiki/FTPS" title="FTPS">FTPS</a> instead of FTP and TelnetS instead of Telnet.</li>
<li>Using a different, more secure protocol that can handle the job, e.g. <a href="/wiki/SSH_File_Transfer_Protocol" title="SSH File Transfer Protocol">SSH File Transfer Protocol</a> or <a href="/wiki/Secure_Copy_Protocol" class="mw-redirect" title="Secure Copy Protocol">Secure Copy Protocol</a>.</li>
<li>Using a secure tunnel such as <a href="/wiki/Secure_Shell" title="Secure Shell">Secure Shell</a> (SSH) or <a href="/wiki/Virtual_private_network" title="Virtual private network">virtual private network</a> (VPN).</li></ol>
<h3><span class="mw-headline" id="FTP_over_SSH">FTP over SSH</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=File_Transfer_Protocol&amp;action=edit&amp;section=11" title="Edit section: FTP over SSH">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>FTP over SSH is the practice of tunneling a normal FTP session over a Secure Shell connection.<sup id="cite_ref-nurdle_17-2" class="reference"><a href="#cite_note-nurdle-17">&#91;17&#93;</a></sup> Because FTP uses multiple <a href="/wiki/Transmission_Control_Protocol" title="Transmission Control Protocol">TCP</a> connections (unusual for a TCP/IP protocol that is still in use), it is particularly difficult to tunnel over SSH. With many SSH clients, attempting to set up a tunnel for the control channel (the initial client-to-server connection on port 21) will protect only that channel; when data is transferred, the FTP software at either end sets up new TCP connections (data channels) and thus have no <a href="/wiki/Confidentiality" title="Confidentiality">confidentiality</a> or <a href="/wiki/Integrity_protection" class="mw-redirect" title="Integrity protection">integrity protection</a>.
</p><p>Otherwise, it is necessary for the SSH client software to have specific knowledge of the FTP protocol, to monitor and rewrite FTP control channel messages and autonomously open new <a href="/wiki/Packet_forwarding" title="Packet forwarding">packet forwardings</a> for FTP data channels. Software packages that support this mode include:
</p>
<ul><li>Tectia ConnectSecure (Win/Linux/Unix)<sup id="cite_ref-18" class="reference"><a href="#cite_note-18">&#91;18&#93;</a></sup> of <a href="/wiki/SSH_Communications_Security" title="SSH Communications Security">SSH Communications Security</a>'s software suite</li></ul>
<h2><span class="mw-headline" id="Derivatives">Derivatives</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=File_Transfer_Protocol&amp;action=edit&amp;section=12" title="Edit section: Derivatives">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<h3><span class="mw-headline" id="FTPS">FTPS</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=File_Transfer_Protocol&amp;action=edit&amp;section=13" title="Edit section: FTPS">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/FTPS" title="FTPS">FTPS</a></div>
<p>Explicit FTPS is an extension to the FTP standard that allows clients to request FTP sessions to be encrypted. This is done by sending the "AUTH TLS" command. The server has the option of allowing or denying connections that do not request TLS. This protocol extension is defined in <a href="/wiki/Request_for_Comments_(identifier)" class="mw-redirect" title="Request for Comments (identifier)">RFC</a>&#160;<a rel="nofollow" class="external text" href="https://tools.ietf.org/html/rfc4217">4217</a>. Implicit FTPS is an outdated standard for FTP that required the use of a SSL or TLS connection. It was specified to use different ports than plain FTP.
</p>
<h3><span class="mw-headline" id="SSH_File_Transfer_Protocol">SSH File Transfer Protocol</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=File_Transfer_Protocol&amp;action=edit&amp;section=14" title="Edit section: SSH File Transfer Protocol">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/SSH_File_Transfer_Protocol" title="SSH File Transfer Protocol">SSH File Transfer Protocol</a></div>
<p>The SSH file transfer protocol (chronologically the second of the two protocols abbreviated SFTP) transfers files and has a similar command set for users, but uses the <a href="/wiki/Secure_Shell" title="Secure Shell">Secure Shell</a> protocol (SSH) to transfer files. Unlike FTP, it encrypts both commands and data, preventing passwords and sensitive information from being transmitted openly over the network. It cannot interoperate with FTP software.
</p>
<h3><span class="mw-headline" id="Trivial_File_Transfer_Protocol">Trivial File Transfer Protocol</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=File_Transfer_Protocol&amp;action=edit&amp;section=15" title="Edit section: Trivial File Transfer Protocol">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Trivial_File_Transfer_Protocol" title="Trivial File Transfer Protocol">Trivial File Transfer Protocol</a></div>
<p>Trivial File Transfer Protocol (TFTP) is a simple, lock-step FTP that allows a client to get a file from or put a file onto a remote host. One of its primary uses is in the early stages of <a href="/wiki/Network_booting" title="Network booting">booting from a local area network</a>, because TFTP is very simple to implement. TFTP lacks security and most of the advanced features offered by more robust file transfer protocols such as File Transfer Protocol. TFTP was first standardized in 1981 and the current specification for the protocol can be found in <a href="/wiki/Request_for_Comments_(identifier)" class="mw-redirect" title="Request for Comments (identifier)">RFC</a>&#160;<a rel="nofollow" class="external text" href="https://tools.ietf.org/html/rfc1350">1350</a>.
</p>
<h3><span class="mw-headline" id="Simple_File_Transfer_Protocol">Simple File Transfer Protocol</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=File_Transfer_Protocol&amp;action=edit&amp;section=16" title="Edit section: Simple File Transfer Protocol">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Simple File Transfer Protocol (the first protocol abbreviated SFTP), as defined by <a href="/wiki/Request_for_Comments_(identifier)" class="mw-redirect" title="Request for Comments (identifier)">RFC</a>&#160;<a rel="nofollow" class="external text" href="https://tools.ietf.org/html/rfc913">913</a>, was proposed as an (unsecured) file transfer protocol with a level of complexity intermediate between TFTP and FTP. It was never widely accepted on the <a href="/wiki/Internet" title="Internet">Internet</a>, and is now assigned Historic status by the <a href="/wiki/Internet_Engineering_Task_Force" title="Internet Engineering Task Force">IETF</a>. It runs through port 115, and often receives the initialism of <i>SFTP</i>. It has a command set of 11 commands and support three types of data transmission: <a href="/wiki/ASCII" title="ASCII">ASCII</a>, <a href="/wiki/Binary_file" title="Binary file">binary</a> and continuous. For systems with a <a href="/wiki/Word_size" class="mw-redirect" title="Word size">word size</a> that is a multiple of 8 bits, the implementation of binary and continuous is the same. The protocol also supports login with user ID and password, hierarchical folders and file management (including <i>rename</i>, <i>delete</i>, <i>upload</i>, <i>download</i>, <i>download with overwrite</i>, and <i>download with append</i>).
</p>

<h2><span class="mw-headline" id="FTP_reply_codes">FTP reply codes</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=File_Transfer_Protocol&amp;action=edit&amp;section=18" title="Edit section: FTP reply codes">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/List_of_FTP_server_return_codes" title="List of FTP server return codes">List of FTP server return codes</a></div>
<p>Below is a summary of <a href="/wiki/List_of_FTP_server_return_codes" title="List of FTP server return codes">FTP reply codes</a> that may be returned by an FTP <a href="/wiki/Server_(computing)" title="Server (computing)">server</a>. These codes have been standardized in <a href="/wiki/Request_for_Comments_(identifier)" class="mw-redirect" title="Request for Comments (identifier)">RFC</a>&#160;<a rel="nofollow" class="external text" href="https://tools.ietf.org/html/rfc959">959</a> by the IETF. The reply code is a three-digit value. The first digit is used to indicate one of three possible outcomes&#160;— success, failure, or to indicate an error or incomplete reply:
</p>
<ul><li>2yz&#160;&#8211; Success reply</li>
<li>4yz or 5yz &#8211; Failure reply</li>
<li>1yz or 3yz &#8211; Error or Incomplete reply</li></ul>
<p>The second digit defines the kind of error:
</p>
<ul><li>x0z &#8211; Syntax. These replies refer to syntax errors.</li>
<li>x1z &#8211; Information. Replies to requests for information.</li>
<li>x2z &#8211; Connections. Replies referring to the control and data connections.</li>
<li>x3z &#8211; Authentication and accounting. Replies for the login process and accounting procedures.</li>
<li>x4z &#8211; Not defined.</li>
<li>x5z &#8211; File system. These replies relay status codes from the server file system.</li></ul>
<p>The third digit of the reply code is used to provide additional detail for each of the categories defined by the second digit.
</p>
