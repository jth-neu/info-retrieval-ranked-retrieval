<h1 id="firstHeading" class="firstHeading" lang="en">Representational state transfer</h1>			<div id="bodyContent" class="mw-body-content">
				<div id="siteSub" class="noprint">From Wikipedia, the free encyclopedia</div>				<div id="contentSub"><span class="mw-redirectedfrom">  (Redirected from <a href="/w/index.php?title=Representational_State_Transfer&amp;redirect=no" class="mw-redirect" title="Representational State Transfer">Representational State Transfer</a>)</span><br />

				<div id="mw-content-text" lang="en" dir="ltr" class="mw-content-ltr"><div class="mw-parser-output"><div role="note" class="hatnote navigation-not-searchable">"REST" redirects here. For other uses, see <a href="/wiki/REST_(disambiguation)" class="mw-redirect mw-disambig" title="REST (disambiguation)">REST (disambiguation)</a>.</div>
<p><b>Representational State Transfer</b> (<b>REST</b>) is an architectural style that defines a set of constraints to be used for creating web services. Web services that conform to the REST architectural style, or <b>RESTful</b> <a href="/wiki/Web_service" title="Web service">web services</a>, provide interoperability between computer systems on the <a href="/wiki/Internet" title="Internet">Internet</a>. REST-compliant web services allow the requesting systems to access and manipulate textual representations of <a href="/wiki/Web_resource" title="Web resource">web resources</a> by using a uniform and predefined set of <a href="/wiki/Stateless_protocol" title="Stateless protocol">stateless</a> operations. Other kinds of web services, such as <a href="/wiki/SOAP" title="SOAP">SOAP</a> web services, expose their own arbitrary sets of operations.<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">&#91;1&#93;</a></sup> 
</p><p>"Web resources" were first defined on the <a href="/wiki/World_Wide_Web" title="World Wide Web">World Wide Web</a> as documents or files identified by their <a href="/wiki/URL" title="URL">URLs</a>. However, today they have a much more generic and abstract definition that encompasses every thing or entity that can be identified, named, addressed, or handled, in any way whatsoever, on the web. In a RESTful web service, requests made to a resource's <a href="/wiki/Uniform_Resource_Identifier" title="Uniform Resource Identifier">URI</a> will elicit a response with a <a href="/wiki/Payload_(computing)" title="Payload (computing)">payload</a> formatted in either <a href="/wiki/HTML" title="HTML">HTML</a>, <a href="/wiki/XML" title="XML">XML</a>, <a href="/wiki/JSON" title="JSON">JSON</a>, or some other format. The response can confirm that some alteration has been made to the stored resource, and the response can provide <a href="/wiki/Hypertext" title="Hypertext">hypertext</a> links to other related resources or collections of resources. When <a href="/wiki/Hypertext_Transfer_Protocol" title="Hypertext Transfer Protocol">HTTP</a> is used, as is most common, the operations available are GET, POST, PUT, DELETE, and other predefined <a href="/wiki/Create,_read,_update_and_delete" title="Create, read, update and delete">CRUD</a> <a href="/wiki/Hypertext_Transfer_Protocol#Request_methods" title="Hypertext Transfer Protocol">HTTP methods</a>. 
</p><p>By using a stateless protocol and standard operations, REST systems aim for fast performance, reliability, and the ability to grow, by re-using components that can be managed and updated without affecting the system as a whole, even while it is running.
</p><p>The term <i>representational state transfer</i> was introduced and defined in 2000 by <a href="/wiki/Roy_Fielding" title="Roy Fielding">Roy Fielding</a> in his doctoral dissertation.<sup id="cite_ref-Fielding-Ch5_2-0" class="reference"><a href="#cite_note-Fielding-Ch5-2">&#91;2&#93;</a></sup><sup id="cite_ref-3" class="reference"><a href="#cite_note-3">&#91;3&#93;</a></sup> Fielding's dissertation explained the REST principles that were known as the "HTTP object model" beginning in 1994, and were used in designing the <a href="/wiki/Hypertext_Transfer_Protocol" title="Hypertext Transfer Protocol">HTTP</a> 1.1 and <a href="/wiki/Uniform_Resource_Identifier" title="Uniform Resource Identifier">Uniform Resource Identifiers</a> (URI) standards.<sup id="cite_ref-4" class="reference"><a href="#cite_note-4">&#91;4&#93;</a></sup><sup id="cite_ref-5" class="reference"><a href="#cite_note-5">&#91;5&#93;</a></sup><sup id="cite_ref-Fielding-Ch6_6-0" class="reference"><a href="#cite_note-Fielding-Ch6-6">&#91;6&#93;</a></sup> The term is intended to evoke an image of how a well-designed Web application behaves: it is a network of Web resources (a virtual state-machine) where the user progresses through the application by selecting links, such as <code>/user/tom</code>, and operations such as GET or DELETE (state transitions), resulting in the next resource (representing the next state of the application) being transferred to the user for their use.
</p>

<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Representational_state_transfer&amp;action=edit&amp;section=1" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<div class="thumb tright"><div class="thumbinner" style="width:222px;"><a href="/wiki/File:Roy_Fielding_at_OSCON_2008.jpg" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/d/d9/Roy_Fielding_at_OSCON_2008.jpg/220px-Roy_Fielding_at_OSCON_2008.jpg" width="220" height="290" class="thumbimage" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/d/d9/Roy_Fielding_at_OSCON_2008.jpg/330px-Roy_Fielding_at_OSCON_2008.jpg 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/d/d9/Roy_Fielding_at_OSCON_2008.jpg/440px-Roy_Fielding_at_OSCON_2008.jpg 2x" data-file-width="2074" data-file-height="2734" /></a>  <div class="thumbcaption"><div class="magnify"><a href="/wiki/File:Roy_Fielding_at_OSCON_2008.jpg" class="internal" title="Enlarge"></a></div>Roy Fielding speaking at <a href="/wiki/O%27Reilly_Open_Source_Convention" title="O&#39;Reilly Open Source Convention">OSCON</a> 2008</div></div></div>
<p><a href="/wiki/Roy_Fielding" title="Roy Fielding">Roy Fielding</a> defined REST in his 2000 PhD dissertation "Architectural Styles and the Design of Network-based Software Architectures" at <a href="/wiki/University_of_California,_Irvine" title="University of California, Irvine">UC Irvine</a>.<sup id="cite_ref-Fielding-Ch5_2-1" class="reference"><a href="#cite_note-Fielding-Ch5-2">&#91;2&#93;</a></sup> He developed the REST architectural style in parallel with <a href="/wiki/Hypertext_Transfer_Protocol" title="Hypertext Transfer Protocol">HTTP</a> 1.1 of 1996–1999, based on the existing design of HTTP 1.0<sup id="cite_ref-Fielding-discuss_7-0" class="reference"><a href="#cite_note-Fielding-discuss-7">&#91;7&#93;</a></sup> of 1996.
</p><p>In a retrospective look at the development of REST, Fielding said: 
</p>

<td style="width: 20px; vertical-align: top; border: none; color: #B2B7F2; font-size: 40px; font-family: &#39;Times New Roman&#39;, Times, serif; font-weight: bold; line-height: .6em; text-align: left; padding: 10px 10px;">“
</td>
<td style="vertical-align: top; border: none; padding: 4px 10px;">Throughout the HTTP standardization process, I was called on to defend the design choices of the Web. That is an extremely difficult thing to do within a process that accepts proposals from anyone on a topic that was rapidly becoming the center of an entire industry. I had comments from well over 500 developers, many of whom were distinguished engineers with decades of experience, and I had to explain everything from the most abstract notions of Web interaction to the finest details of HTTP syntax. That process honed my model down to a core set of principles, properties, and constraints that are now called REST.<sup id="cite_ref-Fielding-discuss_7-1" class="reference"><a href="#cite_note-Fielding-discuss-7">&#91;7&#93;</a></sup>

<h2><span class="mw-headline" id="Architectural_properties">Architectural properties</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Representational_state_transfer&amp;action=edit&amp;section=2" title="Edit section: Architectural properties">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The constraints of the REST architectural style affect the following architectural properties:<sup id="cite_ref-Fielding-Ch5_2-2" class="reference"><a href="#cite_note-Fielding-Ch5-2">&#91;2&#93;</a></sup><sup id="cite_ref-SOA_with_REST_8-0" class="reference"><a href="#cite_note-SOA_with_REST-8">&#91;8&#93;</a></sup>
</p>
<ul><li>performance in component interactions, which can be the dominant factor in user-perceived performance and network efficiency;<sup id="cite_ref-Fielding-Ch2_9-0" class="reference"><a href="#cite_note-Fielding-Ch2-9">&#91;9&#93;</a></sup></li>
<li><a href="/wiki/Scalability" title="Scalability">scalability</a> allowing to support large numbers of components and interactions among components. Roy Fielding describes REST's effect on scalability as follows:</li></ul>
<table class="cquote pullquote" role="presentation" style="margin:auto; border-collapse: collapse; border: none; background-color: transparent; width: auto;">
<tbody><tr>
<td style="width: 20px; vertical-align: top; border: none; color: #B2B7F2; font-size: 40px; font-family: &#39;Times New Roman&#39;, Times, serif; font-weight: bold; line-height: .6em; text-align: left; padding: 10px 10px;">“
</td>
<td style="vertical-align: top; border: none; padding: 4px 10px;">REST's client–server <a href="/wiki/Separation_of_concerns" title="Separation of concerns">separation of concerns</a> simplifies component implementation, reduces the complexity of connector semantics, improves the effectiveness of performance tuning, and increases the scalability of pure server components. Layered system constraints allow intermediaries—<a href="/wiki/Proxy_server" title="Proxy server">proxies</a>, <a href="/wiki/Gateway_(telecommunications)" title="Gateway (telecommunications)">gateways</a>, and <a href="/wiki/Firewall_(computing)" title="Firewall (computing)">firewalls</a>—to be introduced at various points in the communication without changing the interfaces between components, thus allowing them to assist in communication translation or improve performance via large-scale, shared caching. REST enables intermediate processing by constraining messages to be self-descriptive: interaction is stateless between requests, standard methods and media types are used to indicate semantics and exchange information, and responses explicitly indicate <a href="/wiki/Web_cache" title="Web cache">cacheability</a>.<sup id="cite_ref-Fielding-Ch5_2-3" class="reference"><a href="#cite_note-Fielding-Ch5-2">&#91;2&#93;</a></sup>
</td>
<td style="width: 20px; vertical-align: bottom; border: none; color: #B2B7F2; font-size: 40px; font-family: &#39;Times New Roman&#39;, Times, serif; font-weight: bold; line-height: .6em; text-align: right; padding: 10px 10px;">”
</td></tr>

</tbody></table>
<ul><li>simplicity of a uniform interface;</li>
<li>modifiability of components to meet changing needs (even while the application is running);</li>
<li>visibility of communication between components by service agents;</li>
<li>portability of components by moving program code with the data;</li>
<li>reliability in the resistance to failure at the system level in the presence of failures within components, connectors, or data.<sup id="cite_ref-Fielding-Ch2_9-1" class="reference"><a href="#cite_note-Fielding-Ch2-9">&#91;9&#93;</a></sup></li></ul>
<h2><span class="mw-headline" id="Architectural_constraints">Architectural constraints</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Representational_state_transfer&amp;action=edit&amp;section=3" title="Edit section: Architectural constraints">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Six guiding constraints define a RESTful system.<sup id="cite_ref-SOA_with_REST_8-1" class="reference"><a href="#cite_note-SOA_with_REST-8">&#91;8&#93;</a></sup><sup id="cite_ref-Richardson_2007_10-0" class="reference"><a href="#cite_note-Richardson_2007-10">&#91;10&#93;</a></sup> These constraints restrict the ways that the server can process and respond to client requests so that, by operating within these constraints, the service gains desirable <a href="/wiki/Non-functional_requirement" title="Non-functional requirement">non-functional properties</a>, such as performance, scalability, simplicity, modifiability, visibility, portability, and reliability.<sup id="cite_ref-Fielding-Ch5_2-4" class="reference"><a href="#cite_note-Fielding-Ch5-2">&#91;2&#93;</a></sup> If a service violates any of the required constraints, it cannot be considered RESTful.
</p><p>The formal REST constraints are as follows:
</p>
<h3><span id="Client.E2.80.93server_architecture"></span><span class="mw-headline" id="Client–server_architecture">Client–server architecture</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Representational_state_transfer&amp;action=edit&amp;section=4" title="Edit section: Client–server architecture">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div role="note" class="hatnote navigation-not-searchable">See also: <a href="/wiki/Client%E2%80%93server_model" title="Client–server model">Client–server model</a></div>
<p>The principle behind the client–server constraints is the separation of concerns. Separating the user interface concerns from the data storage concerns improves the portability of the user interface across multiple platforms. It also improves scalability by simplifying the server components. Perhaps most significant to the Web, however, is that the separation allows the components to evolve independently, thus supporting the Internet-scale requirement of multiple organizational domains.<sup id="cite_ref-Fielding-Ch5_2-5" class="reference"><a href="#cite_note-Fielding-Ch5-2">&#91;2&#93;</a></sup>
</p>
<h3><span class="mw-headline" id="Statelessness">Statelessness</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Representational_state_transfer&amp;action=edit&amp;section=5" title="Edit section: Statelessness">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div role="note" class="hatnote navigation-not-searchable">See also: <a href="/wiki/Stateless_protocol" title="Stateless protocol">Stateless protocol</a></div>
<p>The client–server communication is constrained by no client context being stored on the server between requests. Each request from any client contains all the information necessary to service the request, and session state is held in the client. The session state can be transferred by the server to another service such as a database to maintain a persistent state for a period and allow authentication. The client begins sending requests when it is ready to make the transition to a new state. While one or more requests are outstanding, the client is considered to be <i>in transition</i>. The representation of each application state contains links that can be used the next time the client chooses to initiate a new state-transition.<sup id="cite_ref-11" class="reference"><a href="#cite_note-11">&#91;11&#93;</a></sup>
</p>
<h3><span class="mw-headline" id="Cacheability">Cacheability</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Representational_state_transfer&amp;action=edit&amp;section=6" title="Edit section: Cacheability">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div role="note" class="hatnote navigation-not-searchable">See also: <a href="/wiki/Web_cache" title="Web cache">Web cache</a></div>
<p>As on the World Wide Web, clients and intermediaries can cache responses. Responses must therefore, implicitly or explicitly, define themselves as cacheable or not to prevent clients from getting stale or inappropriate data in response to further requests. Well-managed caching partially or completely eliminates some client–server interactions, further improving scalability and performance.
</p>
<h3><span class="mw-headline" id="Layered_system">Layered system</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Representational_state_transfer&amp;action=edit&amp;section=7" title="Edit section: Layered system">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div role="note" class="hatnote navigation-not-searchable">See also: <a href="/wiki/Layered_system" title="Layered system">Layered system</a></div>
<p>A client cannot ordinarily tell whether it is connected directly to the end server, or to an intermediary along the way. Intermediary servers can improve system scalability by enabling load balancing and by providing shared caches. They can also enforce security policies.
</p>
<h3><span id="Code_on_demand_.28optional.29"></span><span class="mw-headline" id="Code_on_demand_(optional)">Code on demand (optional)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Representational_state_transfer&amp;action=edit&amp;section=8" title="Edit section: Code on demand (optional)">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<div role="note" class="hatnote navigation-not-searchable">See also: <a href="/wiki/Client-side_scripting" class="mw-redirect" title="Client-side scripting">Client-side scripting</a></div>
<p>Servers can temporarily extend or customize the functionality of a client by transferring executable code. For example, compiled components such as <a href="/wiki/Java_applet" title="Java applet">Java applets</a>, and client-side scripts such as <a href="/wiki/JavaScript" title="JavaScript">JavaScript</a>.
</p>
<h3><span class="mw-headline" id="Uniform_interface">Uniform interface</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Representational_state_transfer&amp;action=edit&amp;section=9" title="Edit section: Uniform interface">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The uniform interface constraint is fundamental to the design of any REST service.<sup id="cite_ref-Fielding-Ch5_2-6" class="reference"><a href="#cite_note-Fielding-Ch5-2">&#91;2&#93;</a></sup> It simplifies and decouples the architecture, which enables each part to evolve independently. The four constraints for this uniform interface are:
</p>
<dl><dt>Resource identification in requests</dt>
<dd>Individual resources are identified in requests, for example using <a href="/wiki/Uniform_resource_identifier" class="mw-redirect" title="Uniform resource identifier">URIs</a> in Web-based REST systems. The resources themselves are conceptually separate from the representations that are returned to the client. For example, the server could send data from its database as <a href="/wiki/HTML" title="HTML">HTML</a>, <a href="/wiki/XML" title="XML">XML</a> or as <a href="/wiki/JSON" title="JSON">JSON</a>—none of which are the server's internal representation.</dd>
<dt>Resource manipulation through representations</dt>
<dd>When a client holds a representation of a resource, including any <a href="/wiki/Metadata" title="Metadata">metadata</a> attached, it has enough information to modify or delete the resource.</dd>
<dt>Self-descriptive messages</dt>
<dd>Each message includes enough information to describe how to process the message. For example, which parser to invoke can be specified by a <a href="/wiki/Media_type" title="Media type">media type</a>.<sup id="cite_ref-Fielding-Ch5_2-7" class="reference"><a href="#cite_note-Fielding-Ch5-2">&#91;2&#93;</a></sup></dd>
<dt>Hypermedia as the engine of application state (<a href="/wiki/HATEOAS" title="HATEOAS">HATEOAS</a>)</dt>
<dd>Having accessed an initial URI for the REST application—analogous to a human Web user accessing the <a href="/wiki/Home_page" title="Home page">home page</a> of a website—a REST client should then be able to use server-provided links dynamically to discover all the available actions and resources it needs. As access proceeds, the server responds with text that includes <a href="/wiki/Hyperlink" title="Hyperlink">hyperlinks</a> to other actions that are currently available. There is no need for the client to be hard-coded with information regarding the structure or dynamics of the REST service.<sup id="cite_ref-RESTfulAPI.net_12-0" class="reference"><a href="#cite_note-RESTfulAPI.net-12">&#91;12&#93;</a></sup></dd></dl>
<h2><span class="mw-headline" id="Applied_to_Web_services">Applied to Web services</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Representational_state_transfer&amp;action=edit&amp;section=10" title="Edit section: Applied to Web services">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p><span id="Applied_to_web_services"></span>
Web service <a href="/wiki/Application_programming_interface" title="Application programming interface">APIs</a> that adhere to the <a href="#Architectural_constraints">REST architectural constraints</a> are called RESTful APIs.<sup id="cite_ref-13" class="reference"><a href="#cite_note-13">&#91;13&#93;</a></sup> HTTP-based RESTful APIs are defined with the following aspects:<sup id="cite_ref-Richardson_2013_14-0" class="reference"><a href="#cite_note-Richardson_2013-14">&#91;14&#93;</a></sup>
</p>
<ul><li>a base <a href="/wiki/URL" title="URL">URL</a>, such as <code>http://api.example.com/resources</code>;</li>
<li>a <a href="/wiki/Media_type" title="Media type">media type</a> that defines state transition data elements (e.g., Atom, microformats, application/vnd.collection+json,<sup id="cite_ref-Richardson_2013_14-1" class="reference"><a href="#cite_note-Richardson_2013-14">&#91;14&#93;</a></sup><sup class="reference" style="white-space:nowrap;">:<span>91–99</span></sup> etc.). The current representation tells the client how to compose requests for transitions to all the next available application states. This could be as simple as a URL or as complex as a Java applet;<sup id="cite_ref-must_be_hypertext_driven_15-0" class="reference"><a href="#cite_note-must_be_hypertext_driven-15">&#91;15&#93;</a></sup></li>
<li>standard <a href="/wiki/HTTP_method" class="mw-redirect" title="HTTP method">HTTP methods</a> (e.g., OPTIONS, GET, PUT, POST, and DELETE).<sup id="cite_ref-16" class="reference"><a href="#cite_note-16">&#91;16&#93;</a></sup></li></ul>
<h3><span class="mw-headline" id="Relationship_between_URL_and_HTTP_methods">Relationship between URL and HTTP methods</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Representational_state_transfer&amp;action=edit&amp;section=11" title="Edit section: Relationship between URL and HTTP methods">edit</a><span class="mw-editsection-bracket">]</span></span></h3>
<p>The following table shows how HTTP methods are typically used in a RESTful API:
</p>
<table class="wikitable">
<caption>HTTP methods
</caption>
<tbody><tr>
<th>Uniform Resource Locator (URL)</th>
<th>GET</th>
<th>PUT</th>
<th>PATCH</th>
<th>POST</th>
<th>DELETE
</th></tr>
<tr>
<th>Collection, such as <code>https://api.example.com/resources/</code>
</th>
<td><b>List</b> the URIs and perhaps other details of the collection's members.
</td>
<td><b>Replace</b> the entire collection with another collection.
</td>
<td>Not generally used
</td>
<td><b>Create</b> a new entry in the collection. The new entry's URI is assigned automatically and is usually returned by the operation.<sup id="cite_ref-thereisnorightway_17-0" class="reference"><a href="#cite_note-thereisnorightway-17">&#91;17&#93;</a></sup>
</td>
<td><b>Delete</b> the entire collection.
</td></tr>
<tr>
<th>Element, such as <code>https://api.example.com/resources/item17</code>
</th>
<td><b>Retrieve</b> a representation of the addressed member of the collection, expressed in an appropriate Internet media type.
</td>
<td><b>Replace</b> the addressed member of the collection, or if it does not exist, <b>create</b> it.
</td>
<td><b>Update</b> the addressed member of the collection.
</td>
<td>Not generally used. Treat the addressed member as a collection in its own right and <b>create</b> a new entry within it.<sup id="cite_ref-thereisnorightway_17-1" class="reference"><a href="#cite_note-thereisnorightway-17">&#91;17&#93;</a></sup>
</td>
<td><b>Delete</b> the addressed member of the collection.
</td></tr></tbody></table>
<p>The GET method is a <a href="/wiki/Hypertext_Transfer_Protocol#Safe_methods" title="Hypertext Transfer Protocol">safe method</a> (or <a href="https://en.wiktionary.org/wiki/nullipotent" class="extiw" title="wikt:nullipotent"><i>nullipotent</i></a>), meaning that calling it produces no <a href="/wiki/Side_effect_(computer_science)" title="Side effect (computer science)">side-effects</a>: retrieving or accessing a record does not change it. The PUT and DELETE methods are <a href="/wiki/Idempotent#Computer_science_meaning" class="mw-redirect" title="Idempotent">idempotent</a>, meaning that the state of the system exposed by the API is unchanged no matter how many times more than once the same request is repeated.
</p><p>Unlike <a href="/wiki/SOAP" title="SOAP">SOAP</a>-based Web services, there is no "official" standard for RESTful Web APIs. This is because REST is an architectural style, while SOAP is a protocol. REST is not a standard in itself, but RESTful implementations make use of standards, such as <a href="/wiki/Hypertext_Transfer_Protocol" title="Hypertext Transfer Protocol">HTTP</a>, <a href="/wiki/Uniform_Resource_Identifier" title="Uniform Resource Identifier">URI</a>, <a href="/wiki/JSON" title="JSON">JSON</a>, and <a href="/wiki/XML" title="XML">XML</a>. Many developers also describe their APIs as being RESTful, even though these APIs actually don't fulfill all of the architectural constraints described above (especially the uniform interface constraint).<sup id="cite_ref-must_be_hypertext_driven_15-1" class="reference"><a href="#cite_note-must_be_hypertext_driven-15">&#91;15&#93;</a></sup>
</p>
